# 設計の出発点: コア課題から機能へ

> **用途**: 技術面談で「どうやって設計したか」を説明するための資料
> **ポイント**: 機能の羅列ではなく「課題→方針→機能」のストーリーで話す
> **最終更新**: 2025-11-28

---

## 1. コア課題（最上位のゴール）

### 1文で表現

```
「忙しい担任が、生徒のSOSを取り逃さないシステム」
```

### なぜこれがコア課題か

| 現場の現実 | 問題 |
|-----------|------|
| 担任は忙しい | 授業、部活、会議、保護者対応... |
| 1クラス30人以上 | 毎日30件以上の日誌を確認する時間がない |
| 紙の連絡帳 | 見逃し、情報共有困難、検索不可 |

**最悪のシナリオ**: 生徒がSOSを出していたのに、担任が見逃す

**このシステムの存在意義**: それを防ぐ

---

## 2. コア課題から導かれた設計方針

コア課題「SOSを取り逃さない」を解決するために、4つの方針を立てた。

```
コア課題: SOSを取り逃さない
    │
    ├─ 方針1: 自動でSOSを検知する
    │         → 担任が見なくても気づける
    │
    ├─ 方針2: 緊急度で自動分類する
    │         → 重要な生徒から処理できる
    │
    ├─ 方針3: 処理を効率化する
    │         → 忙しくても最低限対応できる
    │
    └─ 方針4: 情報を共有できる
              → 一人で抱え込まない
```

### 方針の導出過程

| コア課題の要素 | 問い | 方針 |
|---------------|------|------|
| SOSを取り逃さない | どうやって見つける？ | 自動検知 |
| SOSを取り逃さない | 大量の中からどう優先する？ | 緊急度分類 |
| 忙しい担任 | 時間がない中でどう処理する？ | 効率化 |
| 忙しい担任 | 一人で対応しきれない場合は？ | 情報共有 |

---

## 3. 各方針から導かれた具体的機能

### 方針1: 自動でSOSを検知する → 早期警告機能

| 検知パターン | 意味 | 実装 |
|-------------|------|------|
| メンタル★1 | 「とても落ち込んでいる」= 即対応必要 | P0（重要）に自動分類 |
| 3日連続低下 | じわじわ悪化 = 早期発見 | P1（要注意）に自動分類 |

**なぜこの基準か**:
- ★1は最低評価 → 生徒が「助けて」と言っている
- 3日連続は研究に基づく早期発見の目安

### 方針2: 緊急度で自動分類する → Inbox Pattern

```
P0: 重要      → メンタル★1（赤色）    → 即対応
P1: 要注意    → 3日連続低下（黄色）   → 当日中
P1.5: 要対応  → タスク化済み（橙色）  → 計画的に
P2: 未対応    → 通常の未読（青色）    → 時間があれば
P3: 完了      → 処理済み（緑色）      → 確認不要
```

**なぜInbox Patternか**:
- Gmail/Trelloのように「優先度順」に処理できる
- 医療のトリアージ（緊急度分類）と同じ発想
- 「全部見る」ではなく「重要なものから見る」

### 方針3: 処理を効率化する → ワンクリック処理

| 機能 | 目的 |
|------|------|
| ワンクリック既読 | 特に問題ない日誌は即完了 |
| カード形式表示 | 一覧で状況把握、詳細画面不要 |
| タスク化ボタン | 対応が必要な場合は別管理 |

**なぜワンクリックか**:
- 30人 × 毎日 = 月600回の操作
- 1操作あたり数秒の短縮でも月単位で効果

### 方針4: 情報を共有できる → 共有メモ機能

| 機能 | 目的 |
|------|------|
| 担任メモ | 長期観察記録、引継ぎ |
| 学年共有メモ | 学年会議で情報共有 |
| 学年主任ダッシュボード | 学年全体の傾向把握 |

**なぜ共有か**:
- 担任一人で抱え込むとバーンアウト
- 学年全体で生徒を支える体制

## 4. この思考プロセスの意味

### コア課題から始めると何が良いか

```
✅ コア課題から始めた場合:
   コア課題 → 方針 → 機能

   ・全ての機能に「なぜ必要か」の理由がある
   ・優先順位が明確（コア課題に近いほど重要）
   ・一貫性のある設計になる
   ・「あれもこれも」にならない

❌ 機能から始めた場合:
   機能A, 機能B, 機能C...

   ・「なぜこの機能？」に答えられない
   ・優先順位が決められない
   ・一貫性がない（思いつきの寄せ集め）
   ・「あれもこれも」状態になりやすい
```

### 具体例: 機能追加の判断

```
新機能案: 「日誌にいいねボタンをつけたい」

コア課題で判断:
Q: これは「SOSを取り逃さない」に貢献するか？
A: 直接は貢献しない（あれば便利だが必須ではない）
→ 優先度: 低

新機能案: 「メンタル★1の生徒にアラート通知」

コア課題で判断:
Q: これは「SOSを取り逃さない」に貢献するか？
A: 直接貢献する（見逃し防止）
→ 優先度: 高
```

**コア課題があれば、判断基準が明確になる。**

## 5. 面談での説明例

### 30秒版（エレベーターピッチ）

> 「このシステムのコア課題は『忙しい担任が生徒のSOSを取り逃さない』ことです。そのために、メンタル★1や3日連続低下を自動検知し、緊急度順に分類するInbox Patternを実装しました。これにより、担任は最も重要な生徒から対応できます。」

### 1分版

> 「連絡帳のDX化を行いました。単なるデジタル化ではなく、『忙しい担任がSOSを取り逃さない』というコア課題を設定しました。
>
> 現場では、担任が毎日30人以上の日誌を確認する時間がなく、SOSを見逃すリスクがあります。
>
> これを解決するために、4つの方針を立てました。自動検知、緊急度分類、効率化、情報共有です。
>
> 具体的には、メンタル★1の生徒を赤色で最優先表示し、3日連続低下も自動検知します。これをInbox Patternで優先度順に並べることで、担任は『何から対応すべきか』が一目で分かります。」

### 3分版（詳細説明用）

> 「連絡帳管理システムを設計・開発しました。
>
> **まずコア課題を定義しました。**
> 中学校の担任は非常に忙しく、授業、部活、会議、保護者対応と業務が多いです。その中で毎日30人以上の連絡帳を確認するのは困難です。紙の連絡帳では見逃しが発生しやすく、最悪の場合、生徒がSOSを出していたのに気づかないリスクがあります。
>
> このシステムの存在意義を『忙しい担任がSOSを取り逃さない』と定義しました。
>
> **次に、コア課題を解決する方針を立てました。**
> 4つの柱があります。
> 1つ目は自動検知。メンタル★1や3日連続低下を自動で検出します。
> 2つ目は緊急度分類。Inbox Patternを採用し、P0からP3の5段階で自動分類します。
> 3つ目は効率化。ワンクリックで既読処理ができ、カード形式で一覧把握できます。
> 4つ目は情報共有。学年共有メモで担任間の情報共有を促進します。
>
> **この設計の特徴は、全ての機能がコア課題に紐づいていることです。**
> 例えば『なぜInbox Patternを採用したか』と聞かれたら、『忙しい担任が重要な生徒から対応できるようにするため』と明確に答えられます。コア課題がブレなければ、機能の優先順位も判断できます。」

## 6. 想定質問と回答

### Q: なぜ「日誌のデジタル化」ではなく「SOSを取り逃さない」をコア課題にしたのですか？

**A**: デジタル化は手段であって目的ではないからです。

「日誌をデジタル化する」だけでは、何を優先すべきか判断できません。一方「SOSを取り逃さない」と定義すれば、「この機能はSOSの検知に貢献するか？」という判断基準ができます。

実際、この定義があったから、単なる日誌入力システムではなく、早期警告機能やInbox Patternという設計が生まれました。

### Q: コア課題はどうやって見つけましたか？

**A**: 「最悪のシナリオ」を考えました。

このシステムがなかったら何が起きるか？ → 担任がSOSを見逃す
それを防ぐにはどうすればいいか？ → 自動検知と優先度分類

つまり「防ぎたい最悪の事態」からコア課題を導きました。

### Q: 方針は最初から4つでしたか？

**A**: 最初は「自動検知」と「効率化」の2つでした。

開発を進める中で「情報共有」の重要性に気づき、方針を追加しました。ただし、コア課題「SOSを取り逃さない」はブレていません。方針や機能は調整可能ですが、コア課題は変えないことが重要です。

### Q: この思考プロセスは5D Frameworkですか？

**A**: 5Dの「Define」ステップに相当します。

5D Frameworkでは最初に「誰が何をしたいか」を定義しますが、さらに上位の「なぜこのシステムが存在するか」というコア課題を先に固めました。コア課題が明確なら、ユーザーストーリーも一貫性を持って書けます。

## 7. まとめ: 伝えたいこと

### 私の設計アプローチ

```
1. コア課題を1文で定義する
   → 「SOSを取り逃さない」

2. コア課題を解決する方針を立てる
   → 自動検知、緊急度分類、効率化、情報共有

3. 各方針を実現する機能を設計する
   → Inbox Pattern、早期警告、ワンクリック処理

4. 全ての機能がコア課題に紐づいていることを確認する
   → 「なぜこの機能？」に答えられる状態
```

### この思考プロセスの価値

| 観点 | 価値 |
|------|------|
| **判断基準** | コア課題があれば優先順位が決められる |
| **一貫性** | 全機能がコア課題に紐づき、設計がブレない |
| **説明力** | 「なぜこの機能か」を論理的に説明できる |
| **拡張性** | 新機能もコア課題で判断できる |

### 技術力 + 問題解決力

```
技術力: コードが書ける（当然）
設計力: 構造を考えられる（重要）
問題解決力: 何を作るべきか分かる（最重要）

このシステムでは:
- 現場の課題を理解し（担任は忙しい）
- 本質的な問題を特定し（SOS見逃し）
- 解決策を設計した（Inbox Pattern）
```
