# テスト戦略書

> **対象システム**: 連絡帳管理システム（school_diary）
> **バージョン**: v0.3.0-map
> **作成日**: 2025-10-23
> **目的**: インターン評価向けテスト実施計画

---

## 目的

本ドキュメントは、連絡帳管理システムの品質を保証するためのテスト戦略を定義する。
インターン評価において「テストをしっかり実施した」ことを証明するため、Risk-Based Testing + BDD形式を採用する。

---

## テストレベル（4階層）

### Level 1: Unit Test（単体テスト）

- **対象**: 個別の関数・メソッド
- **ツール**: pytest
- **自動化**: ✅ 完了（150テスト）
- **実行頻度**: コミット時
- **責任者**: 開発者

### Level 2: Integration Test（統合テスト）

- **対象**: 複数コンポーネントの連携（Model + View、Service + Model）
- **ツール**: pytest + Django TestCase
- **自動化**: ✅ 完了（150テストに含まれる）
- **実行頻度**: コミット時
- **責任者**: 開発者

### Level 3: System Test（システムテスト/機能テスト）

- **対象**: ブラウザでの実際の動作確認（画面・機能単位）
- **ツール**: 手動テスト（BDD形式テストケース）
- **自動化**: ⏳ 将来的にPlaywright導入予定
- **実行頻度**: リリース前
- **責任者**: QAエンジニア
- **規模**: 30ケース（Critical 20 + High 10）

### Level 4: Acceptance Test（受入テスト）

- **対象**: ビジネスフロー全体（エンドツーエンド）
- **ツール**: 手動テスト（E2Eシナリオ）
- **自動化**: ⏳ 将来的にPlaywright導入予定
- **実行頻度**: リリース前
- **責任者**: QAエンジニア + ステークホルダー
- **規模**: 5シナリオ

## テストピラミッド

```
         /\
        /AT\ ← 5本のE2Eシナリオ（Acceptance Test）
       /----\
      /  ST  \ ← 30ケースの機能テスト（System Test）
     /--------\
    /    IT    \ ← pytest 150テスト（Integration Test）
   /------------\
  /      UT      \ ← pytest内に含まれる（Unit Test）
 /________________\
```

**方針**: 下層ほど多くのテスト、上層ほど少ないテスト（効率化）

## Risk-Based Testing（優先順位付け）

### Priority分類基準

| Priority | 定義 | 選定基準 |
|----------|------|---------|
| **Critical** | システムの根幹機能、動かないと使用不可 | ログイン、連絡帳作成、ダッシュボード表示 |
| **High** | 主要機能、動かないと価値が低下 | メモ機能、統計機能、早期警告システム |
| **Medium** | 補助機能、動かなくても他で代替可能 | パスワード

リセット、About |
| **Low** | Nice to have、ユーザビリティ向上 | UI改善、誤字修正 |

## BDD形式（Behavior-Driven Development）

### Given-When-Then形式

テストケースは以下の形式で記述する:

```
Feature: 機能名
  役割として、目的を達成したい

Scenario: シナリオ名（正常系/異常系）
  Given 前提条件（ユーザー状態、データ状態）
  And 追加の前提条件
  When 操作（ユーザーアクション）
  And 追加の操作
  Then 期待結果（表示、データ変更）
  And 追加の期待結果
```

### 記述例

```
Feature: 連絡帳作成機能
  生徒として、前日の振り返りを提出したい

Scenario: 生徒が連絡帳を新規作成（正常系）
  Given 生徒アカウント（student_1_a_01@example.com）でログイン済み
  And 今日の連絡帳が未提出
  When 「連絡帳を書く」ボタンをクリック
  And 体調「良い（4）」、メンタル「元気（5）」、振り返り「楽しかった」を入力
  And 「提出する」ボタンをクリック
  Then 「提出しました」メッセージが表示される
  And ダッシュボードに連絡帳が表示される
  And 担任ダッシュボードに「未読1件」が表示される
```

### BDD形式の利点

1. **非エンジニアでも理解できる**: 評価者が読みやすい
2. **再現可能**: 手順が明確、誰でも実行可能
3. **自動化への転換が容易**: 将来Playwright導入時にそのまま使える
4. **仕様書としても機能**: テストケース = 仕様書

## E2Eシナリオ（Acceptance Test）

### 5本のシナリオ

| No | シナリオ名 | 所要時間 | 優先度 |
|----|-----------|----------|--------|
| 1 | 通常フロー（Happy Path） | 5分 | Critical |
| 2 | 早期警告フロー | 10分 | Critical |
| 3 | 担任間共有フロー | 8分 | High |
| 4 | 欠席記録フロー | 7分 | High |
| 5 | 管理者運用フロー | 10分 | Critical |

**合計**: 40分（実際は再テスト含めて1時間）

### シナリオ例

```
Scenario 1: 通常フロー（Happy Path）
1. 生徒が連絡帳を提出
2. 担任がダッシュボードで未読を確認
3. 担任が既読処理
4. 生徒ダッシュボードに反応が表示される
```

## テスト実行記録

### 記録フォーマット

| Step | Expected | Actual | Status | Evidence |
|------|----------|--------|--------|----------|
| 1. ログイン | ダッシュボード表示 | OK | ✅ Pass | ss-001.png |
| 2. 作成画面 | フォーム表示 | OK | ✅ Pass | ss-002.png |
| 3. 入力 | 入力可能 | OK | ✅ Pass | ss-003.png |
| 4. 提出 | 成功メッセージ | エラー表示 | ❌ Fail | ss-004.png |

**Status**:
- ✅ Pass: 期待通り動作
- ❌ Fail: 期待と異なる動作（バグ）
- ⏸️ Blocked: テスト実施不可（環境問題、前提条件不足）
- ⏭️ Skipped: 優先度低下により省略

## Known Issues（既知の問題）の扱い

### 原則

**バグを隠さない。誠実に開示する。**

多くの学生は「動きます」としか言わないが、Known Issuesを記録することで:
- 誠実性をアピール
- 品質意識の高さを示す
- 将来の保守性向上

### 分類

| 重要度 | 定義 | 対応 |
|--------|------|------|
| **Critical** | システムが使えない | 即修正（テスト中断） |
| **High** | 主要機能が動かない | Issue記録、修正は期限次第 |
| **Medium** | 一部機能不良 | Known Issueとして記録、「修正予定」明記 |
| **Low** | UI崩れ、誤字 | Known Issueとして記録、「将来対応」明記 |

### 記録フォーマット

```markdown
## Known Issues

### Issue #1: 連絡帳編集時のバリデーションエラー [Medium]
- **発見日**: 2025-10-24
- **影響範囲**: 生徒の連絡帳編集機能（STU-003）
- **現象**: 既読前に編集しようとするとバリデーションエラー
- **回避策**: 連絡帳を削除して新規作成
- **修正予定**: 2025-10-30
- **テストケース**: TC-STU-003-02
```
