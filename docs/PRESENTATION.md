## 1. 制作の背景と設計意図

### 全体の軸: 担任が生徒の SOS を見逃さない仕組み

紙の連絡帳を DX 化する上で軸に置いたのは「担任の先生が生徒の SOS を見逃さない仕組み」。担任は忙しい中で 30 名以上の生徒を見ており、メンタル面で問題を抱えている生徒を見逃してしまうリスクがある。

この軸を実現するため、基本機能だけで終わらず、実運用を意識した拡張を見越して設計した。

### 拡張を見越した設計

連絡帳の記載・閲覧・既読処理だけでなく、将来的な拡張を見据えて以下を設計した。

**ロールベース権限管理**: 担任だけでなく、学年主任・校長も使えるよう 5 ロールを設計した。各ロールで見える情報の粒度を変え、担任は詳細、学年主任は学年全体、校長は学校全体を確認できるようにした。

**データモデル設計**: 連絡帳だけでなく、出席記録、担任メモ、既読管理など、将来必要になる機能を見越してデータモデルを設計した。

### 一番苦労したのはデザイン設計

技術的な実装は、AI や Web でフォローされるのでそこまで困らなかった。一番苦労したのは「デザイン設計」。

初めてアプリを作ったので、ただ機能を盛り込むだけでは使いづらく、DX 化する意味がないと考えた。この DX で何を実現したいか、どうやったらユーザーが使いやすい形で体験できるか、シンプルにしつつ、意味をどう持たせて処理するかを試行錯誤で何度も繰り返した。

### なぜ Terraform で本番環境を作ったか

アプリケーション本体だけでなく、公開環境まで含めて一貫した制作物にしたかったため、クラウド実装 + IaC を採用した。

クラウドプラットフォームは、自分が知見・経験のある AWS を選択した。限られた時間の中で完成度を優先し、習熟度の高い基盤を使う判断をした。また、AWS には無料枠がありコストを抑えやすい点も決め手となった。

IaC については、インフラをコードで再現可能にし、手作業を減らすために Terraform を選択した。

**補足資料**: [TECHNICAL_SPECIFICATION.md](./TECHNICAL_SPECIFICATION.md)、[PRODUCTION_DEPLOYMENT.md](./PRODUCTION_DEPLOYMENT.md)、[DEPLOYMENT_REFERENCE.md](./DEPLOYMENT_REFERENCE.md)

## 2. 主要機能と改善ポイント

### ユーザー視点で見た改善ポイント

担任・生徒・学年主任それぞれの使い方を想定し、具体的な改善機能を実装した。

**担任の要望**:

- 「時系列でこの生徒の状態がどうなってるか把握したい」
- 「全体感も把握できたらいいな」
- 「担任間で共有したい生徒の状況をタイムリーに共有したい」

**生徒の要望**:

- 「毎日『体調：ふつう』って書いてるかも。数字で表した方がいいのかな?」

**学年主任の要望**:

- 「問題が起こりやすいクラスに早めに気づけるようになるといいな」

### Inbox Pattern による優先度別分類

担任が重要度が高いものから対応できるよう、生徒を 5 段階（6 カテゴリ）に自動分類した。時間がない中で全生徒のページを開いて確認する作業を削減するため、カード化して、連絡帳の内容を一覧で確認し、既読・タスク化を処理できるようにした。

| カテゴリ           | 優先度 | 説明                                 |
| ------------------ | ------ | ------------------------------------ |
| P0: 重要           | 最高   | メンタル ★1（即時対応）              |
| P1: 要注意         | 高     | 3 日連続メンタル低下、（注意が必要） |
| P1.5: 要対応タスク | 高     | 対応記録が設定され、対応待ちの連絡帳 |
| P2: 未対応         | 中     | 日常タスク                           |
| └ P2-1: 未提出     | 中     | 昨日の連絡帳未提出                   |
| └ P2-2: 未読       | 中     | 提出済みだが未確認                   |
| P3: 完了済み       | 低     | 既読・反応済み                       |

### 早期警告システム（3 日連続メンタル低下検知）

担任の力量により、対応が必要なのに対応できず、問題を抱え込んでしまうケースを防ぐため、メンタルで問題がある状態が 3 日続いたときにダッシュボード上で alert として可視化する仕組みを作った。起きていることが早めに見えることで、早期発見・早期対応につなげやすくした。

### 時系列推移表示

担任のヒアリング「時系列で把握したい」に対応するため、各カードに過去 3 日分の体調・メンタル推移を表示した（例: 体調: ★4 → ★3 → ★2 / メンタル: ★3 → ★2 → ★1）。担任は推移を見て「メンタルが下がり続けている」ことに気づける。

### 担任メモの共有機能

担任のヒアリング「タイムリーに共有したい」に対応するため、担任メモを学年内で共有できる機能を実装した。会議の場を待たずに、気づいたときに情報共有できる。

### 数値化（★1-5）

生徒のヒアリング「数字で表した方がいいのかな?」に対応するため、体調・メンタルを ★1-5 で表現した。これにより、推移の変化が一目で分かる。

## 3. 制作で重視した点

### 企画と実装をつなぐこと

実装そのものだけでなく、「このアプリで何を実現したいか」「誰のどの負担を減らしたいか」を先に定めることを重視した。機能追加も、その軸に沿って判断している。

### 使いやすさを優先すること

最も時間を使ったのは UX・デザインで、単に機能を増やすのではなく、忙しい担任でも状況を把握しやすい見せ方を繰り返し調整した。

### 公開環境まで含めて完結させること

ローカル実装だけで終わらせず、AWS と Terraform を使って公開環境まで含めて形にした。アプリ、インフラ、運用導線をまとめて確認できる状態にすることを重視した。
