# Kits パッケージ ドキュメント標準

**バージョン**: 2.0.0
**最終更新**: 2025-10-05
**目的**: 初心者が超一流のエンジニアの実装をトレースし、その背景を理解できるようにする

---

## 🎯 ドキュメントの目的

**最優先目標**: 初心者である読者が、実装の背景・判断理由・思考プロセスを理解できること

**ドキュメントが答えるべき問い**:
1. なぜこの実装が必要だったのか？
2. なぜこの設計を選んだのか？他にどんな選択肢があったのか？
3. どういう順序で実装したのか？なぜその順序なのか？
4. どう使えばいいのか？実際のシナリオは？
5. よくあるつまずきポイントは？どう解決するのか？

---

## ✅ 品質基準（内容重視）

### 1. 設計判断の説明（最重要）

`02_設計思想.md` で以下が**必ず**説明されていること：

✅ **各主要設計判断について**:
- なぜその設計を選んだのか
- 他にどんな選択肢を検討したのか
- 選ばなかった理由は何か
- トレードオフは何か（何を得て、何を失ったか）

❌ **悪い例**: "Djangoモデルを使いました"
✅ **良い例**:
```markdown
### 判断1: Django ORMを使う vs 生SQLを使う

**選択**: Django ORM

**理由**:
- マイグレーション自動生成で変更管理が容易
- 他のkitsパッケージとの一貫性
- 初心者が読みやすい

**検討した代替案**:
- 生SQL: パフォーマンスは高いが、保守性が低い
- SQLAlchemy: 高機能だが、Django標準から外れる

**トレードオフ**:
- 得たもの: 保守性、一貫性、学習コスト低減
- 失ったもの: 複雑なクエリでのパフォーマンス最適化の柔軟性
```

### 2. 実装の流れの説明

`00_実装ログ.md` で以下が説明されていること：

- フェーズごとに何を実装したか
- なぜその順序で実装したか
- 各フェーズでどんな判断をしたか
- どんな問題に直面し、どう解決したか

### 3. 初心者が理解できる説明

`01_概要と目的.md` で以下が説明されていること：

- 具体的な業務シナリオ（初心者がイメージできる）
- このパッケージがない場合の問題点
- このパッケージがある場合の改善点
- 前提知識の明示

### 4. 実践的なコード例

`05_使い方ガイド.md` で以下が提供されていること：

- コピペで動くコード例（10個以上推奨）
- 実際の業務シナリオに基づく例
- 段階的な複雑さ（簡単→応用）
- コード例ごとの「なぜこう書くのか」の説明

### 5. つまずきポイントの先回り

`06_よくある質問.md` で以下が説明されていること：

- 初心者が必ずつまずくポイント
- エラーメッセージとその解決法
- よくある誤解とその訂正
- ベストプラクティス

## 📏 量的な目安（あくまで参考値）

以下は参考値であり、**内容が充実していれば行数は問わない**：

| ドキュメント | 参考行数 | 備考 |
|-------------|---------|------|
| README.md | 200行程度 | ナビゲーションが明確なら短くてもよい |
| 00_実装ログ.md | 300-500行 | 実装規模による |
| 01_概要と目的.md | 200-400行 | 具体例が豊富なら長くなる |
| 02_設計思想.md | **400行以上推奨** | **最重要、設計判断の説明が充実すべき** |
| 03_実装の全体像.md | 300-600行 | コード構造の複雑さによる |
| 05_使い方ガイド.md | 400-700行 | コード例の数による |
| 06_よくある質問.md | 300-600行 | FAQ数による |
| IMPLEMENTATION_SUMMARY.md | 200行程度 | 簡潔でよい |

**重要**: 行数は目安。初心者が理解できるかどうかが判断基準。

## 🚨 品質問題の検出と対応

### 問題検出時

以下のような兆候があれば、ドキュメント改善が必要：

1. **理解度の問題**: 初心者が読んで理解できない
2. **説明不足**: 「なぜ」の説明がない
3. **代替案の欠如**: 他の選択肢を検討していない
4. **つまずき**: 同じ質問が繰り返される

### 改善アプローチ

1. **問題箇所の特定**: どこが理解しづらいか具体的に
2. **根本原因**: なぜ理解しづらいのか（専門用語？説明不足？）
3. **改善策**: 具体例を追加、図を追加、段階的説明など
4. **検証**: 初心者に再度読んでもらう
