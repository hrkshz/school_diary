# kits.reports - 設計思想

**読了時間:** 約10分

このドキュメントは、kits.reportsの設計判断とその理由を記録したものです。
「なぜこの設計にしたのか」「他の選択肢は何だったか」を初心者にも分かるように説明します。

---

## なぜこの設計にしたのか？

### 設計原則

#### 1. **再利用性を最優先**
**内容:** ReportTemplateとReportを分離
**理由:**
- 1つのテンプレートから無限にレポート生成可能
- テンプレート変更で過去のレポートに影響しない
- 異なるパラメータで同じレポートを何度も生成

**実装例:**
```python
# テンプレート = 「型」
template = ReportTemplate.objects.get(code='monthly_overtime')

# レポート = 「インスタンス」
report1 = generate_report(template, parameters={'month': '2025-10'})
report2 = generate_report(template, parameters={'month': '2025-11'})
```

---

#### 2. **セキュリティを重視**
**内容:** UUIDをプライマリキーに採用
**理由:** URL推測攻撃を防ぐ
**実装例:**
```python
# ❌ 連番ID(AutoField)の場合
/reports/1/  # 推測可能
/reports/2/  # 順番に試せる
/reports/3/

# ✅ UUID の場合
/reports/a1b2c3d4-e5f6-7890-abcd-ef1234567890/  # 推測不可能
```

---

#### 3. **ストレージ管理を考慮**
**内容:** 有効期限(expires_at)の導入
**理由:**
- レポートファイルは容量が大きい(平均1-50MB)
- 古いレポートを自動削除してストレージ節約
- RETENTION_DAYS=30で30日後に自動削除

**トレードオフ:**
- ❌ 有効期限なし: ストレージが無限に増加
- ✅ 有効期限あり: 定期クリーンアップ必要だが管理可能

---

#### 4. **状態管理の明確化**
**内容:** ステータス(pending/generating/completed/failed)
**理由:** レポート生成は時間がかかる処理のため

**状態遷移:**
```
pending → generating → completed
          ↓ (エラー時)
        failed
```

**実装例:**
```python
# 生成開始
report.status = 'generating'
report.save()

try:
    # 生成処理
    file_path = export_to_pdf(data)
    report.mark_as_completed(file_path, file_size, row_count)
except Exception as e:
    report.mark_as_failed(str(e))
```

## データモデル設計

### モデル1: ReportTemplate

**役割:** 再利用可能なレポート「型」を管理

**主要フィールドの設計判断:**

| フィールド | 型 | なぜこの型？ | 他の選択肢 | 判断理由 |
|----------|---|------------|----------|---------|
| code | CharField(unique=True) | システム識別子 | AutoField | コードでテンプレート識別(例: monthly_overtime) |
| chart_config | JSONField | 柔軟な設定 | TextField | Chart.js設定を構造化して保存 |
| supported_formats | ArrayField | 複数形式対応 | CharField | PDF, CSV, Excelを同時サポート |
| model_name | CharField | 文字列でモデル指定 | ForeignKey | 動的にモデルを切り替え可能 |

**重要な設計判断1: なぜJSONFieldを使うのか？**

**選択肢の比較:**

| 方式 | メリット | デメリット | 判断 |
|-----|---------|----------|------|
| **JSONField** | 柔軟、構造化、検索可 | PostgreSQL限定 | ✅ 採用 |
| TextField(JSON文字列) | DB非依存 | 検索不可、構造なし | ❌ |
| 別テーブル | 正規化 | 複雑、JOIN必要 | ❌ |

**採用理由:**
- Chart.js設定は複雑な構造(ネスト深い)
- 動的に設定を変更する可能性あり
- PostgreSQLを使うschool_diaryでは最適

**実装例:**
```python
chart_config = {
    'type': 'line',
    'data': {
        'labels': ['1月', '2月', '3月'],
        'datasets': [{
            'label': '売上',
            'data': [100, 200, 150],
        }],
    },
    'options': {
        'responsive': True,
    },
}
```

---

**重要な設計判断2: なぜArrayFieldを使うのか？**

**問題:** 1つのテンプレートで複数形式(PDF, CSV, Excel)をサポートしたい

**選択肢の比較:**

| 方式 | メリット | デメリット | 判断 |
|-----|---------|----------|------|
| **ArrayField** | シンプル、クエリ可 | PostgreSQL限定 | ✅ 採用 |
| ManyToMany | 正規化 | 複雑、JOIN必要 | ❌ |
| CharField(カンマ区切り) | DB非依存 | クエリ困難 | ❌ |

**採用理由:**
- supported_formatsはシンプルなリスト
- 「PDF対応のテンプレート一覧」のクエリが簡単
- PostgreSQLを使うため問題なし

**実装例:**
```python
# テンプレート作成
template.supported_formats = ['pdf', 'csv', 'xlsx']

# クエリ例
pdf_templates = ReportTemplate.objects.filter(
    supported_formats__contains=['pdf']
)
```

### モデル2: Report

**役割:** 個々のレポート生成結果を管理

**主要フィールドの設計判断:**

| フィールド | 型 | なぜこの型？ | 他の選択肢 | 判断理由 |
|----------|---|------------|----------|---------|
| id | UUIDField(PK) | URL推測防止 | AutoField | セキュリティ重視 |
| template | ForeignKey(SET_NULL) | テンプレート削除時も保持 | CASCADE | レポート履歴を残す |
| file | FileField | Djangoファイル管理 | CharField(パス) | メディアファイル標準 |
| parameters | JSONField | 柔軟なパラメータ | TextField | 構造化データ保存 |
| expires_at | DateTimeField | 自動削除 | なし | ストレージ管理 |

**重要な設計判断1: なぜUUIDをPKに？**

**問題:** レポートURLからの推測攻撃を防ぎたい

**選択肢の比較:**

| ID方式 | URL例 | セキュリティ | パフォーマンス | 判断 |
|-------|------|------------|-------------|------|
| **UUIDField** | `/reports/a1b2...ef12/` | 推測不可能 | 若干遅い | ✅ 採用 |
| AutoField | `/reports/1/` | 推測可能 | 高速 | ❌ |
| ランダム文字列 | `/reports/x7kP2q/` | 推測困難 | 衝突リスク | ❌ |

**採用理由:**
- レポートは機密情報を含む可能性
- `/reports/1/`, `/reports/2/` で順番に試せるのは危険
- UUIDなら推測不可能（128bit = 3.4×10³⁸通り）
- パフォーマンスはインデックスで補える

**セキュリティの具体例:**
```python
# ❌ AutoFieldの場合（危険）
http://example.com/reports/1/  # 自分のレポート
http://example.com/reports/2/  # 他人のレポートが見えるかも
http://example.com/reports/3/  # 順番に試せる

# ✅ UUIDの場合（安全）
http://example.com/reports/a1b2c3d4-e5f6-7890-abcd-ef1234567890/
# → 推測不可能、総当たりも現実的でない
```

**重要な設計判断2: なぜtemplate.on_delete=SET_NULL？**

**問題:** テンプレート削除時、レポートをどうするか？

**選択肢の比較:**

| on_delete | 動作 | メリット | デメリット | 判断 |
|----------|------|---------|----------|------|
| **SET_NULL** | テンプレート削除時もレポート残る | 履歴保持 | templateがNone | ✅ 採用 |
| CASCADE | レポートも削除 | データ整合性 | 履歴消失 | ❌ |
| PROTECT | 削除不可 | 安全 | 柔軟性なし | ❌ |

**採用理由:**
- レポート = 「履歴」として価値がある
- テンプレート変更・削除後も過去レポートは必要
- template=Nullでも、titleやfileは残る

**実装例:**
```python
# テンプレート削除
template.delete()

# レポートは残る
report = Report.objects.get(id=uuid)
print(report.template)  # None
print(report.title)     # "月次残業レポート 2025年10月" (残る)
print(report.file.url)  # "/media/reports/..." (残る)
```

**重要な設計判断3: ヘルパーメソッドの配置**

**設計パターン:** ビジネスロジックをモデル層に集約

**実装したメソッド:**
```python
class Report(models.Model):
    # 状態変更メソッド
    def mark_as_completed(self, file_path, file_size, row_count=0):
        """生成完了としてマーク"""
        # 複数フィールドを一括更新

    def mark_as_failed(self, error_message):
        """生成失敗としてマーク"""

    # プロパティ
    @property
    def is_expired(self) -> bool:
        """有効期限切れかどうか"""

    @property
    def is_downloadable(self) -> bool:
        """ダウンロード可能かどうか"""
```

**なぜモデルに配置？**

| 配置場所 | メリット | デメリット | 判断 |
|---------|---------|----------|------|
| **モデル内** | DRY、再利用性 | Fat Model | ✅ 採用 |
| Service層 | 責務分離 | 冗長 | △ 併用 |
| View層 | 簡単 | 重複コード | ❌ |

**採用理由:**
- `mark_as_completed()`は複数箇所で使う
- update_fieldsで最適化（不要なフィールド更新を避ける）
- プロパティで読みやすいコード

### モデル3: ReportSchedule

**役割:** 定期的なレポート生成を管理

**主要フィールドの設計判断:**

| フィールド | 型 | なぜこの型？ | 他の選択肢 | 判断理由 |
|----------|---|------------|----------|---------|
| cron_expression | CharField | Cron形式 | IntegerField(周期) | 柔軟性 |
| send_to_users | ManyToMany | 複数ユーザー | ArrayField | ユーザー管理 |
| send_to_emails | ArrayField | 外部アドレス | TextField | 簡潔 |

**重要な設計判断: なぜCron形式？**

**選択肢の比較:**

| スケジュール方式 | 例 | 柔軟性 | 学習コスト | 判断 |
|--------------|---|-------|-----------|------|
| **Cron形式** | `0 9 * * 1` | 高 | 中 | ✅ 採用 |
| interval(秒) | `86400` (1日) | 低 | 低 | ❌ |
| 独自DSL | `every_monday_9am` | 中 | 高 | ❌ |

**採用理由:**
- Cron = 業界標準（LinuxやCelery Beatで使用）
- 柔軟: 「毎週月曜9時」「毎月1日」など表現可能
- Celery Beatと統合しやすい

**Cron式の例:**
```python
# 毎週月曜日9:00AM
cron_expression = '0 9 * * 1'

# 毎月1日9:00AM
cron_expression = '0 9 1 * *'

# 毎日12:00PM
cron_expression = '0 12 * * *'
```

## アーキテクチャ設計

### レイヤー構成

```
┌─────────────────┐
│   View層        │ ← ユーザーリクエスト受付
└─────────────────┘
         ↓
┌─────────────────┐
│  Service層      │ ← ビジネスロジック（ReportService）
└─────────────────┘
         ↓
┌─────────────────┐
│  Exporter層     │ ← 形式変換（PDFExporter, CSVExporter）
└─────────────────┘
         ↓
┌─────────────────┐
│   Model層       │ ← データ永続化（Report, ReportTemplate）
└─────────────────┘
```

**なぜこの構成？**

**理由1: 責務分離**
- View: リクエスト処理のみ
- Service: レポート生成ロジック
- Exporter: 形式変換のみ
- Model: データ管理のみ

**理由2: テスト容易性**
- 各層を独立してテスト可能
- Mockを使った単体テスト

**理由3: 拡張性**
- 新しい形式追加 → Exporterだけ追加
- 新しいグラフ → ChartBuilderだけ追加

## 技術選定

### 選定1: Chart.js vs Plotly

**問題:** グラフライブラリの選定

**選択肢の比較:**

| ライブラリ | メリット | デメリット | 判断 |
|----------|---------|----------|------|
| **Chart.js** | CDN、軽量、簡単 | 高度機能少 | ✅ 採用 |
| Plotly | 高機能、3Dグラフ | 重い、複雑 | ❌ |
| D3.js | 最強の柔軟性 | 学習コスト高 | ❌ |

**採用理由:**
- Chart.js = CDN利用でnpmビルド不要
- シンプルなAPI（Pythonラッパー実装しやすい）
- 業務アプリには十分な機能

---

### 選定2: WeasyPrint vs ReportLab

**問題:** PDF生成エンジンの選定

**選択肢の比較:**

| エンジン | メリット | デメリット | 判断 |
|---------|---------|----------|------|
| **WeasyPrint** | HTML→PDF、CSS対応 | システム依存 | ✅ 主採用 |
| ReportLab | Python純正、軽量 | HTMLなし | ✅ 代替 |

**両方サポート:**
```python
# 設定で切り替え
REPORTS_CONFIG = {
    'PDF_BACKEND': 'weasyprint',  # or 'reportlab'
}
```

**採用理由:**
- WeasyPrint: Djangoテンプレート→PDFが簡単
- ReportLab: EC2で依存ライブラリなしで動作
- 環境に応じて切り替え可能

## 設計パターン

### パターン1: ビルダーパターン（ChartBuilder）

**問題:** Chart.js設定が複雑すぎる

**解決:** ビルダーパターンで簡潔に

**実装例:**
```python
# ❌ 直接JSON書く場合（複雑）
chart_config = {
    'type': 'line',
    'data': {
        'labels': ['1月', '2月', '3月'],
        'datasets': [{
            'label': '売上',
            'data': [100, 200, 150],
            'borderColor': '#4F46E5',
            'fill': False,
        }],
    },
    'options': {
        'responsive': True,
        'plugins': {
            'title': {'display': True, 'text': '月次売上'},
        },
    },
}

# ✅ ビルダーパターン（簡潔）
chart = ChartBuilder.line(
    labels=['1月', '2月', '3月'],
    datasets=[{'label': '売上', 'data': [100, 200, 150]}],
    title='月次売上',
)
```

---

### パターン2: ストラテジーパターン（Exporter）

**問題:** PDF, CSV, Excel で処理が異なる

**解決:** 共通インターフェースで統一

**実装例:**
```python
# 共通インターフェース
class BaseExporter:
    def export(df, file_path): ...

# 各実装
class PDFExporter(BaseExporter): ...
class CSVExporter(BaseExporter): ...
class ExcelExporter(BaseExporter): ...

# 使用
exporter = get_exporter(format)  # formatに応じて選択
file_size = exporter.export(df, file_path)
```

## トレードオフと将来の改善

### トレードオフ1: PostgreSQL依存

**現状:** ArrayField, JSONFieldを使用（PostgreSQL限定）
**理由:** school_diaryはPostgreSQLを使うため問題なし
**将来:** 他DBサポート時は別フィールドに切り替え

### トレードオフ2: WeasyPrintのシステム依存

**現状:** libpango等のシステムライブラリが必要
**理由:** HTML→PDF変換には必須
**将来:** ReportLabへの完全移行も検討可能

### トレードオフ3: 大量データの処理

**現状:** pandas全件読み込み（メモリ消費）
**理由:** シンプルな実装を優先
**将来:** チャンク処理・ストリーミング対応
