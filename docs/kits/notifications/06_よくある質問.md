## 🔴 エラー対応

### エラー1: NotificationTemplate.DoesNotExist

**エラーメッセージ**:
```
NotificationTemplate matching query does not exist.
```

**原因**:
指定したテンプレートコードが存在しない

**具体例**:
```python
service.create_from_template(
    template_code='welcome_email',  # ← このコードのテンプレートがない
    recipient=user,
    context={}
)
```

**解決方法**:

**方法1: テンプレートを作成**
```python
from kits.notifications.models import NotificationTemplate, NotificationType

NotificationTemplate.objects.create(
    code='welcome_email',  # ← このコードで作成
    name='ウェルカムメール',
    subject_template='ようこそ！',
    body_template='登録ありがとうございます。',
    notification_types=[NotificationType.EMAIL],
)
```

**方法2: コードの typo を確認**
```python
# ❌ typo
template_code='welcom_email'  # ← "e" が抜けている

# ✅ 正しい
template_code='welcome_email'
```

**方法3: 管理画面で確認**
```
http://localhost:8000/admin/notifications/notificationtemplate/
→ 登録されているテンプレートコードを確認
```

### エラー2: TypeError: Object of type User is not JSON serializable

**エラーメッセージ**:
```
TypeError: Object of type User is not JSON serializable
```

**原因**:
contextにDjangoモデルインスタンスを含めたまま保存しようとした

**具体例**:
```python
# ❌ 悪い例
notification = service.create_from_template(
    template_code='approval_request',
    recipient=user,
    context={
        'user': user,  # ← Userオブジェクトを含む（JSONに変換できない）
    }
)
```

**解決方法**:

kits.notificationsは**自動的に**Djangoモデルを除外するので、通常このエラーは発生しません。

ただし、古いバージョンや手動でcontext_dataを保存する場合は注意：

```python
# ✅ 良い例: 必要な値だけ渡す
notification = service.create_from_template(
    template_code='approval_request',
    recipient=user,
    context={
        'user_name': user.get_full_name(),  # ← 文字列で渡す
        'user_email': user.email,            # ← 文字列で渡す
    }
)

# または、Djangoモデルを渡してもOK（自動的に除外される）
notification = service.create_from_template(
    template_code='approval_request',
    recipient=user,
    context={
        'user': user,  # ← 自動的にcontext_dataからは除外される
    }
)
# ただし、テンプレートレンダリング時には使える：
# 件名: "{{user.get_full_name}}さん" → "太郎さん"
```

**services.pyのコード（自動除外の仕組み）**:
```python
# JSONシリアライズ可能なcontextを作成（Djangoモデルを除外）
serializable_context = {
    k: v for k, v in context.items()
    if not hasattr(v, '_meta')  # ← _meta属性を持つ = Djangoモデル
}
notification.context_data = serializable_context
```

### エラー3: 通知が送信されない

**症状**:
- `create_from_template()`は成功する
- ステータスが"pending"のまま
- メールが届かない

**原因と解決方法**:

#### 原因1: Celeryワーカーが起動していない

**確認方法**:
```bash
# Celeryワーカーのログを確認
docker-compose -f docker-compose.local.yml logs celeryworker
```

**解決方法**:
```bash
# Celeryワーカーを起動
docker-compose -f docker-compose.local.yml up celeryworker
```

#### 原因2: 同期送信を使っている

```python
# これだけでは送信されない（通知を作成しただけ）
notification = service.create_from_template(...)

# 解決方法1: 非同期送信
send_notification_task.delay(notification.id)

# 解決方法2: 同期送信
service.send_notification(notification)
```

#### 原因3: scheduled_atが未来

```python
# 将来の日時を設定している場合、その時刻まで送信されない
notification.scheduled_at = datetime(2025, 12, 31)
notification.save()

# 確認方法
print(notification.scheduled_at)
# → 2025-12-31 00:00:00 （未来なのでまだ送信されない）

# 解決方法: scheduled_atをNoneまたは過去に設定
notification.scheduled_at = None
notification.save()
```

### エラー4: SMTPAuthenticationError

**エラーメッセージ**:
```
SMTPAuthenticationError: (535, b'5.7.8 Username and Password not accepted')
```

**原因**:
メールサーバーの認証情報が間違っている

**解決方法**:

**開発環境の場合**:

config/settings/local.py`を使用している場合、メールはコンソールに出力されます：

```python
# config/settings/local.py
EMAIL_BACKEND = 'django.core.mail.backends.console.EmailBackend'

# メールはコンソールログに表示される
# docker-compose logs django で確認
```

**本番環境の場合**:

`config/settings/production.py`のメール設定を確認：

```python
# Amazon SESを使う場合
EMAIL_BACKEND = "anymail.backends.amazon_ses.EmailBackend"
ANYMAIL = {
    "AMAZON_SES_CLIENT_PARAMS": {
        "aws_access_key_id": env("AWS_ACCESS_KEY_ID"),
        "aws_secret_access_key": env("AWS_SECRET_ACCESS_KEY"),
        "region_name": env("AWS_SES_REGION", default="ap-northeast-1"),
    },
}
```

環境変数が正しく設定されているか確認：
```bash
# .env ファイル
AWS_ACCESS_KEY_ID=your_access_key
AWS_SECRET_ACCESS_KEY=your_secret_key
```

### エラー5: テストが失敗する（get_unread_count）

**エラーメッセージ**:
```
AssertionError: 0 != 3
```

**原因**:
通知を作成しただけで、`mark_as_sent()`を呼んでいない

**具体例**:
```python
# ❌ 悪いテストコード
def test_unread_count():
    # 通知を3件作成（ステータス=PENDING）
    for i in range(3):
        Notification.objects.create(
            recipient=user,
            subject=f"Test {i}",
            body="...",
            # status=PENDING （デフォルト）
        )

    # 未読数を取得
    count = service.get_unread_count(user)
    assert count == 3  # ← 失敗！（count=0）
```

**理由**:
`get_unread_count()`は`status=SENT`の通知のみカウントします。
`PENDING`（送信待ち）は未読にカウントされません。

**解決方法**:
```python
# ✅ 正しいテストコード
def test_unread_count():
    # 通知を3件作成
    for i in range(3):
        notification = Notification.objects.create(
            recipient=user,
            subject=f"Test {i}",
            body="...",
        )
        notification.mark_as_sent()  # ← 送信済みにする

    # 未読数を取得
    count = service.get_unread_count(user)
    assert count == 3  # ← 成功！
```

## 💡 よくある質問

### Q1: テンプレートでif文は使える？

**A:** はい、Django Template言語の全機能が使えます。

```django
{% if user.is_superuser %}
**重要**: あなたは管理者権限を持っています。
{% endif %}

{% if request.urgent %}
🚨 **緊急**: この申請は緊急対応が必要です。
{% else %}
通常の申請です。
{% endif %}
```

### Q2: 複数の宛先に同じメールを送りたい

**A:** ループで個別に送信します。

```python
recipients = User.objects.filter(is_active=True)

for user in recipients:
    notification = service.create_from_template(
        template_code='announcement',
        recipient=user,  # ← 個別に作成
        context={'announcement': '重要なお知らせ'}
    )
    send_notification_task.delay(notification.id)
```

**一括送信も可能**:
```python
notifications = []
for user in recipients:
    notification = service.create_from_template(...)
    notifications.append(notification)

result = service.send_batch(notifications)
```

### Q3: HTMLメールを送りたい

**A:** Markdownを使うと自動的にHTMLメールになります。

```python
# テンプレート（Markdown）
body_template = """
# 見出し

これは**太字**です。

- リスト1
- リスト2
"""

# 自動的にHTMLメールに変換される
```

**または、html_templateを使う**:
```python
template = NotificationTemplate.objects.create(
    code='custom_html',
    subject_template='件名',
    body_template='プレーンテキスト版',
    html_template='<h1>見出し</h1><p>HTMLメール</p>',  # ← HTML版
)
```

### Q4: 添付ファイルを送りたい

**A:** 現在のkits.notificationsは添付ファイル非対応です。

**代替案**:

**方法1: URLを送る**
```python
context = {
    'file_url': 'https://school_diary.example.com/files/report.pdf',
}
# 本文: "レポートは [こちら]({{file_url}}) からダウンロードできます。"
```

**方法2: カスタムタスクを作る**
```python
# apps/yourapp/tasks.py
from celery import shared_task
from django.core.mail import EmailMessage

@shared_task
def send_email_with_attachment(recipient_email, subject, body, file_path):
    email = EmailMessage(subject, body, to=[recipient_email])
    email.attach_file(file_path)
    email.send()
```

### Q5: ユーザーが通知設定をOFF/ONできるようにしたい

**A:** カスタムロジックを追加します。

```python
# models.py
class UserNotificationPreference(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    email_enabled = models.BooleanField(default=True)
    in_app_enabled = models.BooleanField(default=True)

# 送信前にチェック
def send_notification_if_enabled(user, template_code, context):
    try:
        pref = user.usernotificationpreference
        if not pref.email_enabled:
            return None  # 通知設定OFF
    except UserNotificationPreference.DoesNotExist:
        pass  # 設定なし = 送信する

    return service.create_from_template(template_code, user, context)
```

### Q6: 送信失敗した通知を再送したい

**A:** 管理画面またはコマンドから再送できます。

**方法1: retry_failed_notificationsタスク**
```python
from kits.notifications.tasks import retry_failed_notifications

# 失敗した通知を全て再送
retry_failed_notifications.delay()
```

**方法2: 個別に再送**
```python
from kits.notifications.models import Notification, NotificationStatus

# 失敗した通知を取得
failed = Notification.objects.filter(status=NotificationStatus.FAILED)

for notification in failed:
    if notification.can_retry:
        send_notification_task.delay(notification.id)
```

### Q7: 通知の送信履歴を削除したい

**A:** cleanup_old_notificationsタスクを使います。

```python
from kits.notifications.tasks import cleanup_old_notifications

# 30日以上前の通知を削除
cleanup_old_notifications.delay(days=30)
```

**手動削除**:
```python
from datetime import timedelta
from django.utils import timezone

# 30日以上前の通知を削除
cutoff_date = timezone.now() - timedelta(days=30)
Notification.objects.filter(created_at__lt=cutoff_date).delete()
```

```python
from kits.notifications.models import Notification

notification = Notification.objects.get(id=123)

print(f"ステータス: {notification.status}")
```

```bash

# Celery Beatのログ（スケジュール送信）
docker-compose -f docker-compose.local.yml logs -f celerybeat
```

### メールが送信されたか確認（開発環境）

```bash
# Djangoログを確認（console backend使用時）
docker-compose -f docker-compose.local.yml logs -f django

# メール内容がコンソールに出力される
```

### データベースを直接確認

```sql
-- 送信待ちの通知
SELECT id, subject, recipient_id, status, created_at
FROM kits_notifications
WHERE status = 'pending'
ORDER BY created_at DESC;

-- 送信失敗の通知
SELECT id, subject, error_message, retry_count
FROM kits_notifications
WHERE status = 'failed'
ORDER BY created_at DESC;

-- 未読通知の数（ユーザーID=1）
SELECT COUNT(*)
FROM kits_notifications
WHERE recipient_id = 1 AND status = 'sent';
```

## ⚡ パフォーマンス改善

### 大量送信時の注意

**悪い例**:
```python
# 1万人に送る
for user in User.objects.all():  # 1万件
    notification = service.create_from_template(...)
    send_notification_task.delay(notification.id)  # 1万回タスク作成😓
```

**良い例**:
```python
# バッチ送信を使う
notifications = []
for user in User.objects.all():
    notification = service.create_from_template(...)
    notifications.append(notification)

# バッチで送信
service.send_batch(notifications, batch_size=100)
```

### データベースクエリの最適化

```python
# ❌ N+1問題
notifications = Notification.objects.filter(recipient=user)
for notification in notifications:
    print(notification.template.name)  # ← 毎回クエリ発行😓

# ✅ select_relatedで最適化
notifications = Notification.objects.filter(
    recipient=user
).select_related('template', 'recipient')
for notification in notifications:
    print(notification.template.name)  # ← クエリ1回😊
```
