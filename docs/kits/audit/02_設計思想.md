## 🎯 設計の大原則

kits.auditは以下の3つの原則に基づいて設計されています：

### 1. **シンプルさ優先** 🎨
複雑な構造を避け、ファイル数を最小限に保つ

### 2. **疎結合** 🔗
他のアプリケーションに依存しない、再利用可能な設計

### 3. **Django標準に従う** 📘
Djangoの慣習やベストプラクティスに準拠

## 📐 設計判断の記録

### 判断1: 2層構造の監査システム

#### 選択した設計

```python
# レイヤー1: django-simple-historyによる自動履歴
class AuditMixin(models.Model):
    history = HistoricalRecords()  # フィールドレベルの変更を自動記録

# レイヤー2: カスタム監査ログ
class AuditLog(models.Model):
    event_type = models.CharField(...)  # ビジネスイベント
    changes = models.JSONField(...)     # 変更内容
    metadata = models.JSONField(...)    # 追加情報
```

#### なぜこの設計？

**理由1: それぞれの強みを活かす**

```
django-simple-historyの強み:
✅ フィールドレベルの変更を自動追跡
✅ 差分の自動計算
✅ ロールバック機能
✅ 設定が簡単（1行追加するだけ）

django-simple-historyの弱点:
❌ ビジネスイベント（承認、却下など）を記録できない
❌ 追加メタデータ（IPアドレスなど）を保存できない
❌ 複数モデルにまたがる操作を1つのイベントとして記録できない

カスタムAuditLogの強み:
✅ ビジネスレベルのイベント記録
✅ 任意のメタデータ保存
✅ 横断的な監査レポート生成

カスタムAuditLogの弱点:
❌ フィールドレベルの変更は手動実装が必要
❌ 差分計算を自分で実装する必要がある
```

**理由2: 実際の使用シーンでの使い分け**

```python
# シーン1: 「誰がいつ金額を変更したか」知りたい
# → django-simple-historyで自動記録される
overtime.amount = 5000
overtime._history_user = request.user
overtime.save()

# シーン2: 「誰がいつ承認したか」記録したい
# → カスタムAuditLogで記録
audit_service.log_event(
    event_type="approve",
    event_name="残業申請承認",
    obj=overtime,
    user=manager,
    metadata={"approval_reason": "妥当な理由"}
)
```

#### 他の選択肢は？

**選択肢A: django-simple-historyのみ（却下）**
```python
# すべてdjango-simple-historyで対応
class OvertimeRequest(models.Model):
    history = HistoricalRecords()
    # 承認イベントも無理やりフィールド変更で表現
    approval_event = models.CharField(...)  # これは不自然
```
❌ 却下理由:
- ビジネスイベントをフィールド変更で表現するのは不自然
- メタデータの保存場所がない
- イベントとフィールド変更が混在して分かりにくい

**選択肢B: カスタムAuditLogのみ（却下）**
```python
# すべて手動で記録
def save_overtime(overtime, user):
    old_value = OvertimeRequest.objects.get(pk=overtime.pk)
    # 全フィールドの差分を手動計算...😰
    changes = calculate_changes(old_value, overtime)
    audit_log.create(changes=changes, ...)
```
❌ 却下理由:
- フィールドレベルの追跡が大変
- 差分計算のバグリスク
- 車輪の再発明

#### トレードオフ

**得たもの**:
- ✅ 適材適所の使い分け
- ✅ 両方の利点を獲得
- ✅ 実装の簡潔さ

**失ったもの**:
- ❌ 2つのシステムを理解する学習コスト
- ❌ データが2箇所に分散
- ❌ 若干のストレージオーバーヘッド

### 判断2: AuditMixinの継承アプローチ

#### 選択した設計

```python
class AuditMixin(models.Model):
    history = HistoricalRecords(inherit=True)

    class Meta:
        abstract = True  # 抽象クラスとして定義
```

#### なぜこの設計？

**理由1: 各モデルが独自のhistoryテーブルを持つ**

```
OvertimeRequest → historical_overtimerequest テーブル
BookLoan → historical_bookloan テーブル
```

利点：
- パフォーマンスが良い（テーブルが分離）
- インデックスが効率的
- マイグレーションが独立

**理由2: 既存モデルへの追加が簡単**

```python
# Before: 履歴なし
class OvertimeRequest(models.Model):
    hours = models.IntegerField()

# After: 1行追加するだけ
class OvertimeRequest(AuditMixin):  # ← これだけ！
    hours = models.IntegerField()
```

#### 他の選択肢は？

**選択肢A: 共通historyテーブル（GenericForeignKey使用）（却下）**
```python
class UniversalHistory(models.Model):
    content_type = models.ForeignKey(ContentType)
    object_id = models.CharField()
    content_object = GenericForeignKey()
    # すべてのモデルの履歴を1テーブルに
```
❌ 却下理由:
- GenericForeignKeyはクエリ性能が悪い
- JOINが使えない
- インデックスが効きにくい

**選択肢B: デコレータパターン（却下）**
```python
@audit_enabled
class OvertimeRequest(models.Model):
    pass
```
❌ 却下理由:
- Djangoのモデル継承メカニズムと相性が悪い
- メタクラスの黒魔術が必要
- デバッグが困難

#### トレードオフ

**得たもの**:
- ✅ シンプルで理解しやすい
- ✅ パフォーマンスが良い
- ✅ 既存コードへの影響最小

**失ったもの**:
- ❌ 全モデル横断的な履歴検索が困難
- ❌ テーブル数が増える

### 判断3: サービス層の分離

#### 選択した設計

```python
class AuditService:
    """ビジネスロジックを担当"""
    def log_event(self, ...): pass
    def get_user_activity(self, ...): pass
    def generate_audit_report(self, ...): pass
```

#### なぜこの設計？

**理由1: 責務の分離**

```
Model層: データ構造の定義
Service層: ビジネスロジック（← ここに集約）
View層: リクエスト処理
```

**理由2: テスタビリティ**

```python
# テストが書きやすい
def test_audit_service():
    service = AuditService()
    mock_user = Mock()
    result = service.get_user_activity(mock_user, days=30)
    assert len(result) == expected_count
```

**理由3: 複数モデルを横断する処理**

```python
def generate_audit_report(self, start_date, end_date):
    # AuditLogとHistoricalRecordsの両方から情報を収集
    audit_logs = AuditLog.objects.filter(...)
    history_records = self._collect_all_history(...)
    return self._merge_and_format(audit_logs, history_records)
```

#### 他の選択肢は？

**選択肢A: モデルメソッド（却下）**
```python
class AuditLog(models.Model):
    def generate_report(self):  # モデルが肥大化
        pass
    def send_notification(self):
        pass
    def export_to_pdf(self):
        pass
```
❌ 却下理由:
- モデルの責務過多（Single Responsibility原則違反）
- テストが困難
- 依存関係が複雑化

**選択肢B: カスタムマネージャー（却下）**
```python
class AuditLogManager(models.Manager):
    def generate_report(self):
        pass
```
❌ 却下理由:
- マネージャーはDBアクセスに特化すべき
- ビジネスロジックの置き場所として不適切

#### トレードオフ

**得たもの**:
- ✅ クリーンなアーキテクチャ
- ✅ テストしやすい
- ✅ 再利用しやすい

**失ったもの**:
- ❌ ファイル数が1つ増える
- ❌ 初心者には層が多く見える

### 判断4: JSONFieldでの柔軟なデータ保存

#### 選択した設計

```python
class AuditLog(models.Model):
    changes = models.JSONField(default=dict)    # 変更内容
    metadata = models.JSONField(default=dict)   # メタデータ
```

#### なぜこの設計？

**理由1: スキーマレスで柔軟**

```python
# どんな変更内容でも保存可能
changes = {
    "status": {"from": "draft", "to": "submitted"},
    "amount": {"from": 1000, "to": 1500},
    "custom_field": {"added": "new_value"}
}

# どんなメタデータでも保存可能
metadata = {
    "ip_address": "192.168.1.1",
    "browser": "Chrome",
    "api_version": "v2",
    "custom_context": {...}
}
```

**理由2: 将来の拡張が簡単**

```python
# 新しい要件: 位置情報も記録したい
metadata["location"] = {
    "lat": 35.6812,
    "lng": 139.7671
}
# → マイグレーション不要！
```

#### 他の選択肢は？

**選択肢A: 専用カラム（却下）**
```python
class AuditLog(models.Model):
    old_status = models.CharField(...)
    new_status = models.CharField(...)
    old_amount = models.IntegerField(...)
    new_amount = models.IntegerField(...)
    # 無限に増える...😱
```
❌ 却下理由:
- 事前にすべての変更パターンを予測不可能
- カラムが爆発的に増える
- NULL値だらけのテーブルになる

**選択肢B: EAV（Entity-Attribute-Value）パターン（却下）**
```python
class AuditChange(models.Model):
    audit_log = models.ForeignKey(AuditLog)
    field_name = models.CharField()
    old_value = models.TextField()
    new_value = models.TextField()
```
❌ 却下理由:
- クエリが複雑
- パフォーマンスが悪い
- 型情報が失われる

#### トレードオフ

**得たもの**:
- ✅ 無限の拡張性
- ✅ マイグレーション不要
- ✅ 実装がシンプル

**失ったもの**:
- ❌ SQLでの直接検索が困難
- ❌ 型安全性なし
- ❌ インデックスが効かない

### 判断5: イベントタイプの事前定義

#### 選択した設計

```python
EVENT_TYPE_CHOICES = [
    ("create", "Created"),
    ("update", "Updated"),
    ("delete", "Deleted"),
    ("approve", "Approved"),
    ("reject", "Rejected"),
    ("submit", "Submitted"),
    ("cancel", "Cancelled"),
    ("custom", "Custom"),  # 拡張用
]
```

#### なぜこの設計？

**理由1: 標準化とタイポ防止**

```python
# ❌ タイポのリスク
log_event(event_type="apprve")  # approveのつもり

# ✅ 選択肢から選ぶので安全
log_event(event_type="approve")  # IDEが補完
```

**理由2: 集計・分析が簡単**

```python
# 承認イベントを集計
AuditLog.objects.filter(event_type="approve").count()
# → 確実に全承認イベントを取得
```

**理由3: 管理画面での使いやすさ**

Djangoの管理画面で、ドロップダウンから選択できる

#### 他の選択肢は？

**選択肢A: 完全自由文字列（却下）**
```python
event_type = models.CharField(max_length=100)  # 何でもOK
```
❌ 却下理由:
- "approve", "approved", "承認" が混在
- 集計が困難
- タイポに気づきにくい

**選択肢B: Enumで完全固定（却下）**
```python
class EventType(Enum):
    CREATE = "create"
    UPDATE = "update"
    # 追加不可能...
```
❌ 却下理由:
- 新しいイベントタイプの追加が困難
- コード変更が必要
- 柔軟性がない

#### トレードオフ

**得たもの**:
- ✅ 統一性と拡張性の両立
- ✅ タイポ防止
- ✅ 分析しやすい

**失ったもの**:
- ❌ 完全な自由度
- ❌ 事前定義の手間

## 🔍 設計の全体的なトレードオフ

### 得たもの（メリット）

1. **使いやすさ**: 1行追加で履歴管理が有効化
2. **柔軟性**: どんなモデル、どんな変更にも対応
3. **パフォーマンス**: 適切な分離で高速動作
4. **保守性**: クリーンなコード構造
5. **拡張性**: 将来の要件変更に強い

### 失ったもの（デメリット）

1. **学習コスト**: 2つのシステムを理解する必要
2. **ストレージ**: 若干の重複データ
3. **複雑さ**: 初見では層が多く見える

---

## 📝 まとめ

kits.auditの設計は、**実用性とシンプルさのバランス**を重視しています。

完璧を目指すのではなく、80%のユースケースを20%の複雑さで解決することを目標としました。

結果として：
- ✅ 初心者でも使える（AuditMixin継承するだけ）
- ✅ 上級者も満足（AuditServiceで高度な処理）
- ✅ 将来の変更に強い（JSONFieldで拡張可能）

これらの設計判断により、**実務で本当に使える**監査証跡システムを実現しています。
