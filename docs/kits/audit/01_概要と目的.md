## 🤔 監査証跡（Audit Trail）とは？

### 簡単に言うと

**「誰が、いつ、何を、変更したか」を記録し、後から追跡できるようにする仕組み**です。

### 具体例

#### 例1: 残業申請の変更履歴
```
2025-01-05 10:00: 山田太郎さんが残業申請を作成
2025-01-05 14:00: 山田太郎さんが申請内容を修正（2時間 → 3時間）
2025-01-06 09:00: 鈴木課長が承認
2025-01-06 09:05: 佐藤部長が最終承認
```

**全ての変更が記録され、誰が何をしたか明確**です。

#### 例2: マスターデータの不正変更の検出
```
2025-01-10 23:45: admin_user が商品価格を変更
  変更前: 10,000円
  変更後: 1,000円
```

**深夜の不審な変更を検出**できます。

#### 例3: 個人情報アクセスの追跡
```
2025-01-15 15:30: user123 が顧客ID=456のデータを閲覧
  IPアドレス: 192.168.1.100
  ブラウザ: Chrome 120
```

**個人情報保護法対応のため、アクセスログを記録**します。

## 🎯 なぜ監査証跡が必要なのか？

### 問題1: データが勝手に変更されている

誰が変更したか分からないと、**問題の原因究明ができません**。

```python
# 従来のシステム
class Document(models.Model):
    title = models.CharField(max_length=200)
    updated_at = models.DateTimeField(auto_now=True)
    # ❌ 変更した人、変更理由が記録されない
```

**これでは困ります:**
- 重要なデータが削除されても、誰がやったか分からない
- 不正な変更があっても、証拠が残らない
- トラブル時に原因を追跡できない

### 問題2: 法令・規制への対応

多くの業界で**監査証跡の保持が法的に義務付けられています**。

**対応が必要な分野:**
- 医療: 電子カルテの変更履歴（医療法）
- 金融: 取引記録の保存（金融商品取引法）
- 行政: 公文書の変更履歴（公文書管理法）
- 個人情報: アクセスログの記録（個人情報保護法）

### 問題3: セキュリティインシデントの追跡

**不正アクセスや情報漏洩が発生したとき**、監査証跡がないと：
- いつ侵入されたか分からない
- どのデータが漏洩したか不明
- 被害範囲を特定できない

### 問題4: ユーザーからの問い合わせ対応

「データが消えた」「勝手に変更された」という問い合わせに対して、**証拠を示して説明できません**。

## ✨ kits.auditで解決できること

### 解決1: 自動的な変更履歴記録

```python
from kits.audit.models import AuditMixin

class OvertimeRequest(AuditMixin):
    """残業申請（履歴自動記録）"""
    employee = models.ForeignKey(User)
    hours = models.DecimalField(max_digits=4, decimal_places=2)
    # AuditMixinを継承するだけで、全ての変更が記録される
```

**これだけで:**
- ✅ 誰が変更したか記録される
- ✅ いつ変更したか記録される
- ✅ 変更前後の値が記録される
- ✅ 変更理由を記録できる

### 解決2: カスタムイベントのログ記録

```python
from kits.audit.services import AuditService

service = AuditService()

# ビジネスロジックレベルのイベントを記録
service.log_event(
    event_type="approve",
    event_name="残業申請承認",
    obj=overtime_request,
    user=manager,
    description="課長が承認しました",
    changes={"status": {"from": "pending", "to": "approved"}},
    metadata={"department": "営業部"}
)
```

**これで:**
- ✅ 承認・却下などのイベントを記録
- ✅ 変更内容を構造化して保存
- ✅ 追加情報（部署、優先度等）も記録

### 解決3: 強力な検索とレポート機能

```python
# 過去30日間の特定ユーザーのアクティビティ
activity = service.get_user_activity(user, days=30)

# 今月の監査レポート
report = service.generate_audit_report(
    start_date=datetime(2025, 1, 1),
    end_date=datetime(2025, 1, 31)
)

print(f"総イベント数: {report['total_events']}")
print(f"ユーザー別: {report['user_activity']}")
```

**できること:**
- ✅ ユーザー、期間、イベントタイプで検索
- ✅ 変更履歴の比較
- ✅ 統計レポートの自動生成

### 解決4: 管理画面で履歴を閲覧

**Django Adminで監査ログを閲覧:**
- ✅ 色分けされたイベントタイプ表示
- ✅ 変更前後の値を並べて表示
- ✅ ユーザー情報、IPアドレスも表示
- ✅ 検索・フィルタ機能

## 🔍 どんな時に使うのか？

### 使用例1: 承認フローの監査証跡

```python
# kits.approvalsと組み合わせて使用
approval_request.approve(user=manager)

# 自動的に監査ログが記録される
# - 誰が承認したか
# - いつ承認したか
# - どのステップで承認したか
```

### 使用例2: マスターデータ変更の追跡

```python
# 商品マスターの価格変更
product.price = 15000
product._history_user = request.user
product._change_reason = "メーカー希望小売価格改定"
product.save()

# 履歴から変更を確認
for record in product.history.all():
    print(f"{record.history_date}: {record.price}円")
```

### 使用例3: 個人情報アクセスの記録

```python
# 顧客情報閲覧時にログ記録
service.log_event(
    event_type="custom",
    event_name="顧客情報閲覧",
    obj=customer,
    user=request.user,
    user_ip=request.META.get('REMOTE_ADDR'),
    metadata={"purpose": "問い合わせ対応"}
)
```

### 使用例4: セキュリティ監査

```python
# 深夜のアクセスを検出
night_activity = service.search_logs(
    start_date=datetime.combine(date.today(), time(22, 0)),
    end_date=datetime.combine(date.today(), time(6, 0))
)

if night_activity:
    print("警告: 深夜のアクセスがあります")
```

## 📊 kits.auditの2つの機能

### 機能1: django-simple-history（自動履歴記録）

**全てのモデル変更を自動記録**

- モデルに `history = HistoricalRecords()` を追加するだけ
- または `AuditMixin` を継承
- 全てのCRUD操作が自動的に履歴に記録される

### 機能2: AuditLogモデル（カスタムイベント記録）

**ビジネスロジックレベルのイベントを記録**

- 承認・却下などの高レベルイベント
- カスタムイベントの記録
- IPアドレス、User-Agent等の追加情報
- 柔軟な検索とレポート機能
