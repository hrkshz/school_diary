# 02. 設計思想 - なぜこう設計したのか？

**読了時間: 12分**

## 📖 このドキュメントで学べること

- なぜこの設計を選んだのか？
- 他にどんな方法があったのか？（代替案）
- それぞれのメリット・デメリットは？（トレードオフ）

---

## 🎯 設計の基本方針

kits.approvalsは以下の3つの原則に基づいて設計されています：

1. **汎用性**: どんなモデルにも承認フローを追加できる
2. **柔軟性**: 承認フローを動的に変更できる
3. **保守性**: コードを読みやすく、拡張しやすい

---

## 🔑 主要な設計判断

### 判断1: GenericForeignKeyを使う

#### なぜこの設計を選んだのか？

**承認フローを任意のモデルに適用できるようにするため**です。

```python
class ApprovalRequest(models.Model):
    # GenericForeignKey: どのモデルとも関連付けられる
    content_type = models.ForeignKey(ContentType, on_delete=models.CASCADE)
    object_id = models.PositiveIntegerField()
    content_object = GenericForeignKey("content_type", "object_id")
```

#### 代替案1: 継承（Abstract Base Class）

```python
class ApprovableModel(models.Model):
    status = models.CharField(...)

    class Meta:
        abstract = True

# 使用例
class OvertimeRequest(ApprovableModel):
    pass
```

**メリット**:
- 型安全性が高い
- クエリが高速

**デメリット**:
- 既存のモデルに適用しにくい
- 全てのモデルで継承が必要

#### 代替案2: 各モデルにForeignKey

```python
class ApprovalRequest(models.Model):
    overtime_request = models.ForeignKey(OvertimeRequest, null=True, ...)
    leave_request = models.ForeignKey(LeaveRequest, null=True, ...)
    # ... モデルごとにForeignKeyを追加
```

**メリット**:
- クエリが高速
- リレーションが明確

**デメリット**:
- 新しいモデルごとにマイグレーションが必要
- スケールしない

#### なぜGenericForeignKeyを選んだのか？

| 観点 | GenericForeignKey | 継承 | 各モデルFK |
|------|-------------------|------|------------|
| 汎用性 | ⭐⭐⭐ | ⭐ | ⭐ |
| 既存モデルへの適用 | ⭐⭐⭐ | ⭐ | ⭐⭐ |
| 保守性 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| パフォーマンス | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |

**結論**: 汎用性と保守性を最優先し、パフォーマンスは必要に応じてキャッシュで対応する方針を採用しました。

### 判断2: ワークフローとステップを分離

#### なぜこの設計を選んだのか？

**承認フローを動的に変更できるようにするため**です。

```python
class ApprovalWorkflow(models.Model):
    name = models.CharField(...)
    # ワークフロー全体の設定

class ApprovalStep(models.Model):
    workflow = models.ForeignKey(ApprovalWorkflow, ...)
    order = models.PositiveIntegerField()
    # 個別のステップの設定
```

#### 代替案1: ハードコードされたステップ

```python
class ApprovalRequest(models.Model):
    manager_approved = models.BooleanField(default=False)
    director_approved = models.BooleanField(default=False)
    executive_approved = models.BooleanField(default=False)
```

**メリット**:
- シンプル
- 高速

**デメリット**:
- ステップ数を変更できない
- 承認者のロールを変更できない
- 再利用できない

#### 代替案2: JSONFieldに格納

```python
class ApprovalWorkflow(models.Model):
    steps = models.JSONField(default=list)
    # 例: [{"name": "課長承認", "role": "Manager"}, ...]
```

**メリット**:
- 柔軟
- 追加のテーブルが不要

**デメリット**:
- クエリが複雑
- バリデーションが難しい
- Django管理画面で編集しにくい

#### なぜ分離する設計を選んだのか？

| 観点 | 分離（現設計） | ハードコード | JSON |
|------|---------------|------------|------|
| 柔軟性 | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| クエリ性能 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| 管理画面での操作 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| バリデーション | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |

**結論**: Django ORMの利点を最大限に活用し、管理画面での操作性を重視しました。

### 判断3: サービス層を導入

#### なぜこの設計を選んだのか？

**ビジネスロジックをモデルから分離し、再利用性とテストしやすさを向上させるため**です。

```python
# kits/approvals/services.py
class ApprovalService:
    def approve_step(self, request, step, approver, comment=""):
        # 承認ロジックを一元管理
        # - 権限チェック
        # - 並列承認の処理
        # - 次のステップへの進行
        # - 通知の送信
```

#### 代替案1: Fat Modelパターン

```python
class ApprovalRequest(models.Model):
    def approve(self, step, approver):
        # モデルメソッドで全て処理
```

**メリット**:
- Djangoらしい書き方
- シンプル

**デメリット**:
- モデルが肥大化
- 複数のモデルにまたがる処理が難しい
- テストが複雑

#### 代替案2: ビュー層で直接処理

```python
def approve_view(request):
    # ビューで承認ロジックを実装
    approval_request.current_step = next_step
    approval_request.save()
```

**メリット**:
- 最もシンプル

**デメリット**:
- ロジックが分散
- 再利用できない
- テストが困難

#### なぜサービス層を選んだのか？

| 観点 | サービス層 | Fat Model | ビュー層 |
|------|-----------|-----------|---------|
| 再利用性 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| テストしやすさ | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| 保守性 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| Djangoらしさ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐ |

**結論**: 長期的な保守性とテストしやすさを重視し、サービス層を導入しました。

### 判断4: 並列承認の実装方法

#### なぜこの設計を選んだのか？

**「N人中M人の承認」という柔軟な要件に対応するため**です。

```python
class ApprovalStep(models.Model):
    is_parallel = models.BooleanField(default=False)
    required_approvals = models.PositiveIntegerField(default=1)

# サービス層で判定
if step.is_parallel:
    approval_count = ApprovalAction.objects.filter(
        request=request, step=step, action="approve"
    ).count()

    if approval_count < step.required_approvals:
        return request  # まだ次のステップに進まない
```

#### 代替案1: 全員の承認が必要

```python
# シンプルだが柔軟性がない
approvers = step.approver_role.user_set.all()
if all(user.has_approved(request) for user in approvers):
    # 次のステップへ
```

**メリット**:
- シンプル
- 明確

**デメリット**:
- 「3人中2人」のような柔軟な要件に対応できない

#### 代替案2: 複雑なルールエンジン

```python
class ApprovalStep(models.Model):
    approval_rule = models.JSONField()
    # 例: {"type": "majority", "min_percent": 50}
```

**メリット**:
- 非常に柔軟

**デメリット**:
- 複雑すぎる
- オーバーエンジニアリング
- 初心者には理解しにくい

#### なぜこの設計を選んだのか？

| 観点 | 現設計 | 全員必須 | ルールエンジン |
|------|--------|---------|---------------|
| 柔軟性 | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| シンプルさ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| 理解しやすさ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |

**結論**: 必要十分な柔軟性とシンプルさのバランスを重視しました。

### 判断5: 期限管理の実装

#### なぜこの設計を選んだのか？

**期限切れを自動でチェックし、リマインダーを送信するため**です。

```python
# Celeryタスクで定期実行
@shared_task
def check_overdue_requests():
    overdue_requests = service.get_overdue_requests()
    for request in overdue_requests:
        _send_overdue_notification(request)
```

#### 代替案1: プロパティで判定のみ

```python
class ApprovalRequest(models.Model):
    @property
    def is_overdue(self):
        return self.deadline and timezone.now() > self.deadline
```

**メリット**:
- シンプル
- リアルタイム

**デメリット**:
- 自動通知ができない
- ユーザーがページを開くまで気づかない

#### 代替案2: Djangoシグナルで即座に通知

```python
@receiver(post_save, sender=ApprovalRequest)
def check_deadline(sender, instance, **kwargs):
    if instance.is_overdue():
        send_notification()
```

**メリット**:
- 即座に反応

**デメリット**:
- 保存のたびにチェックが走る（非効率）
- 期限切れになった瞬間には反応できない

#### なぜCeleryタスクを選んだのか？

| 観点 | Celeryタスク | プロパティのみ | シグナル |
|------|-------------|--------------|---------|
| 自動通知 | ⭐⭐⭐ | ⭐ | ⭐⭐ |
| パフォーマンス | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| 定期実行 | ⭐⭐⭐ | ⭐ | ⭐ |

**結論**: 定期的なバッチ処理が最も効率的で信頼性が高いと判断しました。

### 判断6: 自動承認の実装

#### なぜこの設計を選んだのか？

**「申請者が承認者ロールに属している場合は自動で次のステップに進む」という要件に対応するため**です。

```python
class ApprovalStep(models.Model):
    auto_approve_if_requester_in_role = models.BooleanField(default=False)

# 提出時にチェック
if (step.auto_approve_if_requester_in_role
    and step.can_approve(requester)):
    service.auto_approve_step(request, step, requester)
```

#### 代替案1: カスタムロジックを毎回実装

```python
# ユーザーが自分で実装
def submit_request(request):
    if request.requester in manager_group:
        # 課長ステップをスキップ
        request.current_step = director_step
```

**メリット**:
- 完全に自由

**デメリット**:
- 毎回実装が必要
- バグが入りやすい
- 再利用できない

#### 代替案2: 複雑な条件エンジン

```python
class ApprovalStep(models.Model):
    auto_approve_condition = models.JSONField()
    # 例: {"if": "requester.role == approver.role", "then": "auto_approve"}
```

**メリット**:
- 非常に柔軟

**デメリット**:
- オーバーエンジニアリング
- セキュリティリスク
- 理解が困難

#### なぜこの設計を選んだのか？

| 観点 | 現設計 | カスタム実装 | 条件エンジン |
|------|--------|------------|-------------|
| 再利用性 | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| 安全性 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| シンプルさ | ⭐⭐⭐ | ⭐⭐ | ⭐ |

**結論**: 最も一般的な「ロール一致での自動承認」のみを標準機能とし、それ以外はカスタマイズで対応する方針を採用しました。

### 判断7: 通知機能との連携

#### なぜこの設計を選んだのか？

**kits.notificationsがある場合は自動で通知を送り、ない場合でも動作するようにするため**です。

```python
def _send_notification(self, request, template_code, actor):
    try:
        from kits.notifications.services import NotificationService
        service = NotificationService()
        service.create_from_template(...)
    except (ImportError, Exception) as e:
        logger.debug("Could not send notification: %s", e)
```

#### 代替案1: 必須依存関係にする

```python
# kits.notificationsを必ず使う
from kits.notifications.services import NotificationService
```

**メリット**:
- シンプル
- 常に通知が送られる

**デメリット**:
- kits.notificationsがないと動かない
- 疎結合性が失われる

#### 代替案2: シグナルで連携

```python
# Django信号を使う
approval_approved = django.dispatch.Signal()

@receiver(approval_approved)
def send_notification(sender, request, **kwargs):
    # 通知を送信
```

**メリット**:
- 完全に疎結合
- 拡張性が高い

**デメリット**:
- 設定が複雑
- デバッグが難しい

#### なぜtry-exceptを選んだのか？

| 観点 | try-except | 必須依存 | シグナル |
|------|-----------|---------|---------|
| 疎結合性 | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| シンプルさ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| デバッグしやすさ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐ |

**結論**: シンプルさと疎結合性のバランスを重視しました。

## 🎨 アーキテクチャパターン

### レイヤー構造

```
┌─────────────────────────────────┐
│  Django Admin / Views / API     │ ← ユーザーインターフェース
├─────────────────────────────────┤
│  ApprovalService                │ ← ビジネスロジック層
├─────────────────────────────────┤
│  Models (Workflow, Request, ...) │ ← データアクセス層
├─────────────────────────────────┤
│  Database (PostgreSQL)          │ ← データ永続化層
└─────────────────────────────────┘
```

**なぜこの構造？**
- 関心の分離（Separation of Concerns）
- テストしやすさ
- 再利用性

## 📚 設計パターン

### 1. Service Layerパターン

**目的**: ビジネスロジックを一元管理

```python
class ApprovalService:
    def approve_step(self, request, step, approver, comment=""):
        # 複雑なビジネスロジックをここに集約
```

### 2. Strategyパターン（並列承認）

**目的**: 承認方法を柔軟に切り替える

```python
if step.is_parallel:
    # 並列承認のロジック
else:
    # 単一承認のロジック
```

### 3. Template Methodパターン（通知）

**目的**: 通知の送信ロジックを共通化

```python
def _send_notification(self, request, template_code, actor):
    # 共通の通知送信ロジック
```

## 🔄 トレードオフのまとめ

| 設計判断 | 選択 | トレードオフ |
|---------|------|-------------|
| 汎用性 | GenericForeignKey | パフォーマンス < 柔軟性 |
| 動的フロー | Workflow分離 | 複雑さ < 変更容易性 |
| ロジック配置 | Service Layer | Djangoらしさ < テスト容易性 |
| 並列承認 | 単純な数値指定 | 高度な柔軟性 < シンプルさ |
| 期限管理 | Celeryタスク | リアルタイム性 < 効率性 |
| 通知連携 | Optional依存 | 一貫性 < 疎結合性 |

## 💡 学んだ教訓

### 1. YAGNI（You Aren't Gonna Need It）

複雑な承認ルールエンジンは**実装しませんでした**。
必要になったら、その時に追加すれば良いと判断しました。

### 2. 必要十分な抽象化

過度な汎用化は避け、**実際の業務で使われるパターンに焦点を当てました**。

### 3. Django Wayを尊重しつつ、現代的なパターンも取り入れる

DjangoのFat ModelパターンとService Layerパターンの**良いとこ取り**をしました。
