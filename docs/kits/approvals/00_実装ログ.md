### 設計判断

#### 判断1: GenericForeignKeyの採用

**決定**: GenericForeignKeyを使用して、任意のモデルと関連付け可能にする

**理由**:
- どのモデルにも承認フローを追加できる汎用性
- 既存のモデルに簡単に適用可能
- 他のkitsとの一貫性

**代替案**:
- 継承（Abstract Base Class）: 既存モデルへの適用が困難
- 各モデルにForeignKey: スケールしない

#### 判断2: ワークフローとステップの分離

**決定**: ApprovalWorkflowとApprovalStepを別モデルとして実装

**理由**:
- 管理画面で動的に承認フローを変更可能
- 複数のモデルで同じワークフローを再利用可能
- Django ORMの利点を最大限に活用

**代替案**:
- JSONFieldに格納: クエリが複雑、バリデーションが困難
- ハードコード: 柔軟性がない

#### 判断3: サービス層の導入

**決定**: ApprovalServiceクラスでビジネスロジックを管理

**理由**:
- ロジックの一元管理
- テストしやすさ
- 再利用性

**代替案**:
- Fat Model: モデルが肥大化
- ビュー層で処理: ロジックが分散

## 技術的な課題と解決

### 課題1: Userモデルがusernameフィールドを持たない

**問題**:
```python
# school_diaryのUserモデルはemailをUSERNAME_FIELDとしている
User.objects.create_user(username="test", email="test@example.com")
# → TypeError: User() got unexpected keyword arguments: 'username'
```

**解決**:
```python
# emailのみで作成
User.objects.create_user(email="test@example.com", password="password")
```

**影響**: テストコード全体を修正

### 課題2: kits.notificationsのAPIパラメータ

**問題**:
```python
# related_objectパラメータが存在しない
service.create_from_template(..., related_object=request)
# → TypeError
```

**解決**:
```python
# related_object_typeとrelated_object_idに分離
service.create_from_template(
    ...,
    related_object_type="approval_request",
    related_object_id=str(request.id),
)
```

### 課題3: Pylanceの型チェックエラー

**問題**:
- TYPE_CHECKINGパターンでの型ヒント
- 未使用のimport警告

**解決**:
- pyrightconfig.jsonで適切な設定
- TYPE_CHECKINGブロックの活用
