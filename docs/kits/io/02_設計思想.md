## 🎯 設計の大原則

kits.ioは以下の3つの原則に基づいて設計されています：

### 1. **シンプルさ優先** 🎨
複雑な構造を避け、ファイル数を最小限に保つ

### 2. **疎結合** 🔗
他のアプリケーションに依存しない、再利用可能な設計

### 3. **Django標準に従う** 📘
Djangoの慣習やベストプラクティスに準拠

## 📐 設計判断の記録

### 判断1: データモデルの設計

#### 選択した設計

```python
# 2つのモデルで構成
class ImportMapping(models.Model):
    """マッピング設定（再利用可能）"""
    code = models.CharField(...)           # マッピング識別子
    model_name = models.CharField(...)     # 対象モデル
    field_mapping = models.JSONField(...)  # フィールド対応関係
    unique_fields = models.JSONField(...)  # 重複チェック対象
    duplicate_strategy = models.CharField(...)  # 重複処理戦略

class ImportHistory(models.Model):
    """個々のインポート実行結果"""
    mapping = models.ForeignKey(ImportMapping, ...)
    original_filename = models.CharField(...)
    status = models.CharField(...)         # pending/processing/completed/failed
    total_rows = models.PositiveIntegerField(...)
    success_count = models.PositiveIntegerField(...)
    error_details = models.JSONField(...)  # 行ごとのエラー
```

#### なぜこの設計？

**理由1: マッピング設定の再利用**

同じファイル形式を何度もインポートする場合、毎回マッピングを指定するのは面倒です。

```python
# ❌ 悪い例: 毎回マッピングを指定
importer = CSVImporter(
    model_name='library.Book',
    mapping={'タイトル': 'title', '著者': 'author', ...},  # 毎回書く
)

# ✅ 良い例: マッピング設定を再利用
importer = CSVImporter(
    model_name='library.Book',
    mapping_code='book_import',  # コードを指定するだけ
)
```

**理由2: インポート履歴の完全な記録**

- いつ、誰が、何をインポートしたか
- 成功率、エラー内容、処理時間
- 監査証跡として重要

**理由3: 1対多の関係**

1つのマッピング設定から複数のインポート実行が行われる関係を表現。

#### 他の選択肢は？

**選択肢A: 1つのモデルにまとめる**
```python
class Import(models.Model):
    field_mapping = models.JSONField(...)  # マッピング
    status = models.CharField(...)         # ステータス
    success_count = models.IntegerField(...)
```

**却下理由:**
- マッピング設定が再利用できない
- データが冗長になる

### 判断2: UUIDをPKに採用（ImportHistory）

#### 選択した設計

```python
class ImportHistory(models.Model):
    id = models.UUIDField(
        primary_key=True,
        default=uuid.uuid4,
        editable=False,
    )
```

#### なぜこの設計？

**理由1: セキュリティ向上**

連番ID（1, 2, 3...）だと、他のユーザーのインポート履歴を推測できてしまいます。

```python
# ❌ 連番IDの場合
# http://example.com/admin/io/importhistory/1/
# http://example.com/admin/io/importhistory/2/  ← 推測可能

# ✅ UUIDの場合
# http://example.com/admin/io/importhistory/
#   a1b2c3d4-e5f6-7890-abcd-ef1234567890/  ← 推測不可能
```

**理由2: 分散環境でのID衝突回避**

将来的に複数サーバーで並行処理する際、UUIDならID衝突の心配がありません。

**理由3: kits.reportsと設計を統一**

既存のkits.reportsパッケージでも同様の理由でUUIDを採用しており、一貫性を保ちます。

#### 他の選択肢は？

**選択肢A: 連番ID（AutoField）**
```python
id = models.AutoField(primary_key=True)
```

**却下理由:**
- セキュリティリスク
- 分散環境での衝突リスク

### 判断3: ステータス管理の導入

#### 選択した設計

```python
class ImportStatus(models.TextChoices):
    PENDING = "pending", _("処理待ち")
    PROCESSING = "processing", _("処理中")
    COMPLETED = "completed", _("完了")
    FAILED = "failed", _("失敗")
    PARTIAL = "partial", _("一部成功")

# ステータス遷移
pending → processing → completed/failed/partial
```

#### なぜこの設計？

**理由1: 処理の進捗追跡**

大量データのインポートは時間がかかるため、現在の状態を明確にする必要があります。

```python
# ユーザーが確認できる
if history.status == ImportStatus.PROCESSING:
    print("処理中です... しばらくお待ちください")
elif history.status == ImportStatus.COMPLETED:
    print(f"完了しました！ {history.success_count}件")
```

**理由2: エラーハンドリング**

処理が失敗した場合と、一部成功した場合を区別できます。

```python
if history.status == ImportStatus.FAILED:
    print(f"エラー: {history.error_message}")
elif history.status == ImportStatus.PARTIAL:
    print(f"一部成功: 成功{history.success_count}件、失敗{history.failed_count}件")
```

**理由3: 非同期処理への対応**

将来的にCeleryなどで非同期処理する際、ステータス管理が必須です。

#### 他の選択肢は？

**選択肢A: Boolean フラグ**
```python
is_completed = models.BooleanField(default=False)
is_failed = models.BooleanField(default=False)
```

**却下理由:**
- 状態の組み合わせが複雑になる
- 中間状態（processing）を表現できない

### 判断4: 重複処理戦略の柔軟性

#### 選択した設計

```python
class DuplicateStrategy(models.TextChoices):
    SKIP = "skip", _("スキップ")
    UPDATE = "update", _("更新")
    RENUMBER = "renumber", _("新規採番")
    ERROR = "error", _("エラー")
```

#### なぜこの設計？

**理由1: 過去課題への対応**

課題4（図書館システム）では、バーコードIDが重複した際に新規採番が必要でした。

```python
# 課題4の要件
# 5つの学校図書館データを統合
# バーコードIDが重複している場合は新規採番

importer = ExcelImporter(
    model_name='library.Book',
    unique_fields=['barcode_id'],
    duplicate_strategy='renumber',  # これが必要だった
)

# 結果:
# A校: ABC-001 → ABC-001（そのまま）
# B校: ABC-001 → ABC-001_00001（新規採番）
# C校: ABC-001 → ABC-001_00002（新規採番）
```

**理由2: 様々なユースケースに対応**

| 戦略 | 使用場面 | 例 |
|------|----------|-----|
| skip | 既存データを保護したい | マスタデータの初回登録 |
| update | 最新データで上書きしたい | 月次データの更新 |
| renumber | 重複を避けて全て登録したい | 複数拠点のデータ統合 |
| error | 重複を厳格に禁止したい | 一意制約の厳守 |

**理由3: Strategy パターンの採用**

オブジェクト指向の設計パターン「Strategy パターン」を採用し、処理ロジックを切り替え可能にしました。

```python
def _handle_duplicate(self, existing, row_dict):
    if self.duplicate_strategy == DuplicateStrategy.SKIP:
        return 'skipped'
    elif self.duplicate_strategy == DuplicateStrategy.UPDATE:
        # 更新処理
        return 'updated'
    elif self.duplicate_strategy == DuplicateStrategy.RENUMBER:
        # 新規採番処理
        return 'renumbered'
    elif self.duplicate_strategy == DuplicateStrategy.ERROR:
        raise ValidationError("重複データ")
```

#### 他の選択肢は？

**選択肢A: skip のみ実装**
```python
if existing:
    continue  # スキップのみ
```

**却下理由:**
- 過去課題4の要件（新規採番）に対応できない
- 柔軟性に欠ける

### 判断5: チャンク処理の採用

#### 選択した設計

```python
def _import_dataframe(self, df, history):
    # チャンク処理
    for chunk_start in range(0, len(df), self.chunk_size):
        chunk_end = min(chunk_start + self.chunk_size, len(df))
        chunk = df.iloc[chunk_start:chunk_end]

        with transaction.atomic():  # チャンクごとにトランザクション
            for idx, row in chunk.iterrows():
                # 1行ずつ処理
```

#### なぜこの設計？

**理由1: メモリ効率**

100万行のデータを一度にメモリに展開すると、サーバーがクラッシュする可能性があります。

```python
# ❌ 悪い例: 全行を一度に処理
for row in df.iterrows():  # 100万行を一度にメモリ展開
    process(row)

# ✅ 良い例: チャンクごとに処理
for chunk in chunks(df, size=1000):  # 1000行ずつ
    for row in chunk.iterrows():
        process(row)
```

**理由2: トランザクション管理**

チャンクごとにトランザクションをコミットすることで、途中エラーでも処理済み分は保存されます。

```python
# チャンク1 (1-1000行): 成功 → コミット
# チャンク2 (1001-2000行): 成功 → コミット
# チャンク3 (2001-3000行): エラー → ロールバック（チャンク3のみ）
# → 2000行分は保存済み
```

**理由3: プログレス表示**

将来的にプログレスバーを表示する際、チャンク処理が便利です。

```python
total_chunks = len(df) // chunk_size
for i, chunk in enumerate(chunks(df, size=chunk_size)):
    progress = (i + 1) / total_chunks * 100
    print(f"進捗: {progress:.1f}%")
```

#### 他の選択肢は？

**選択肢A: 全行を一度に処理**
```python
for row in df.iterrows():
    process(row)
```

**却下理由:**
- 大量データでメモリ不足
- 途中エラーで全ロールバック

### 判断6: マッピング設定の分離

#### 選択した設計

```python
# マッピング設定（再利用可能）
class ImportMapping(models.Model):
    code = models.CharField(...)
    field_mapping = models.JSONField(...)

# インポート履歴（個別実行）
class ImportHistory(models.Model):
    mapping = models.ForeignKey(ImportMapping, ...)
```

#### なぜこの設計？

**理由1: DRY原則（Don't Repeat Yourself）**

同じマッピングを何度も書かない。

```python
# ❌ 悪い例: 毎回マッピングを指定
# 1回目
importer = CSVImporter(
    mapping={'タイトル': 'title', '著者': 'author', 'ISBN': 'isbn'}
)

# 2回目（同じマッピングをまた書く）
importer = CSVImporter(
    mapping={'タイトル': 'title', '著者': 'author', 'ISBN': 'isbn'}
)

# ✅ 良い例: マッピング設定を再利用
importer = CSVImporter(mapping_code='book_import')  # 簡潔
```

**理由2: 設定の一元管理**

マッピング設定を変更したい場合、1箇所を修正するだけで全体に反映されます。

```python
# マッピングを変更
mapping = ImportMapping.objects.get(code='book_import')
mapping.field_mapping['出版社'] = 'publisher'  # 追加
mapping.save()

# 以降のインポートで自動的に反映される
```

**理由3: 管理画面での設定**

技術者でなくても、Django管理画面からマッピング設定を変更できます。

#### 他の選択肢は？

**選択肢A: 毎回マッピングを指定**
```python
# コードに直接書く
importer = CSVImporter(
    mapping={'タイトル': 'title', ...}
)
```

**却下理由:**
- コードの重複
- 設定変更のたびにコード修正が必要

### 判断7: 文字コード自動検出

#### 選択した設計

```python
if self.auto_detect_encoding:
    with open(file_path, 'rb') as f:
        result = chardet.detect(f.read())
        encoding = result['encoding']  # 自動検出
else:
    encoding = self.encoding  # 手動指定
```

#### なぜこの設計？

**理由1: 現実世界の複雑さに対応**

実務では、様々な文字コードのファイルを扱います。

```
取引先A: UTF-8 のCSV
取引先B: Shift-JIS のCSV（Windowsで作成）
取引先C: EUC-JP のCSV（Unix系で作成）
```

**理由2: ユーザビリティ向上**

文字コードを意識しなくても使えるようにします。

```python
# ✅ 初心者でも簡単
importer = CSVImporter(
    model_name='library.Book',
    auto_detect_encoding=True,  # 自動検出
)
# 文字コードを知らなくてもOK！

# 上級者は明示的に指定も可能
importer = CSVImporter(
    model_name='library.Book',
    encoding='shift-jis',
    auto_detect_encoding=False,
)
```

**理由3: chardetライブラリの活用**

Mozilla開発の高精度な文字コード検出ライブラリを活用。

#### 他の選択肢は？

**選択肢A: UTF-8のみサポート**
```python
df = pd.read_csv(file_path, encoding='utf-8')
```

**却下理由:**
- Shift-JISファイルで文字化け
- 実務での使用に耐えられない

### 判断8: pandasの採用

#### 選択した設計

```python
# pandasでファイル読み込み
df = pd.read_csv(file_path, encoding=encoding)
df = pd.read_excel(file_path, sheet_name=sheet_name)

# DataFrameで統一的に処理
for idx, row in df.iterrows():
    row_dict = row.to_dict()
    # Djangoモデルに保存
```

#### なぜこの設計？

**理由1: CSV/Excel の統一的な処理**

pandasなら、CSV/TSV/Excelを同じDataFrameとして扱えます。

```python
# ✅ pandas使用
df_csv = pd.read_csv('data.csv')
df_tsv = pd.read_csv('data.tsv', delimiter='\t')
df_excel = pd.read_excel('data.xlsx')
# → 全てDataFrameとして統一的に処理

# ❌ 標準ライブラリのみ
import csv
with open('data.csv') as f:
    reader = csv.DictReader(f)
    # CSV専用の処理

import openpyxl
wb = openpyxl.load_workbook('data.xlsx')
# Excel専用の処理（まったく違う）
```

**理由2: データ変換の柔軟性**

DataFrameなら、データ変換が簡単です。

```python
# カラム名変更（マッピング）
df = df.rename(columns={'タイトル': 'title', '著者': 'author'})

# NaN値をNoneに変換
row_dict = row.where(pd.notna(row), None).to_dict()
```

**理由3: 実績のあるライブラリ**

pandasはデータ分析のデファクトスタンダードで、バグが少なく安定しています。

#### 他の選択肢は？

**選択肢A: 標準ライブラリのみ（csv, openpyxl）**
```python
import csv
with open('data.csv') as f:
    reader = csv.DictReader(f)
```

**却下理由:**
- CSV/Excelで処理が分岐
- データ変換が煩雑

## 🎨 設計パターンの採用

### 1. **Strategy パターン**（重複処理）

```python
def _handle_duplicate(self, existing, row_dict):
    # 戦略を切り替え
    if self.duplicate_strategy == DuplicateStrategy.SKIP:
        return self._strategy_skip()
    elif self.duplicate_strategy == DuplicateStrategy.UPDATE:
        return self._strategy_update(existing, row_dict)
    # ...
```

### 2. **Template Method パターン**（インポーター）

```python
class BaseImporter:
    def import_file(self, file_path):
        # テンプレートメソッド
        df = self._read_file(file_path)  # サブクラスで実装
        result = self._import_dataframe(df)
        return result

class CSVImporter(BaseImporter):
    def _read_file(self, file_path):
        # CSV固有の処理
        return pd.read_csv(file_path)
```

### 3. **Builder パターン**（将来拡張）

```python
# 将来的に複雑な設定をBuilderパターンで
importer = (
    ImporterBuilder()
    .for_model('library.Book')
    .with_mapping('book_import')
    .with_validation(enabled=True)
    .build()
)
```

## 🔍 トレードオフの記録

設計には必ずトレードオフがあります。選択しなかった代替案とその理由を記録します。

### トレードオフ1: UUID vs 連番ID

| 項目 | UUID | 連番ID |
|------|------|--------|
| セキュリティ | ✅ 高い | ❌ 低い（推測可能） |
| パフォーマンス | ⚠️ やや遅い | ✅ 速い |
| 分散環境 | ✅ 衝突なし | ❌ 衝突リスク |
| URL | 長い | 短い |

**選択**: UUID（セキュリティを優先）

### トレードオフ2: pandas vs 標準ライブラリ

| 項目 | pandas | 標準ライブラリ |
|------|--------|---------------|
| 統一性 | ✅ CSV/Excel統一 | ❌ 個別処理 |
| 依存関係 | ⚠️ 重い | ✅ 軽い |
| データ変換 | ✅ 簡単 | ❌ 煩雑 |
| 学習コスト | ⚠️ やや高い | ✅ 低い |

**選択**: pandas（統一性と柔軟性を優先）
