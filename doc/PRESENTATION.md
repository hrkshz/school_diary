## 1. 課題 1 の工夫点・アピール

### 全体の軸: 担任が生徒の SOS を見逃さない仕組み

紙の連絡帳を DX 化する上で軸に置いたのは「担任の先生が生徒の SOS を見逃さない仕組み」。担任は忙しい中で 30 名以上の生徒を見ており、メンタル面で問題を抱えている生徒を見逃してしまうリスクがある。

この軸を実現するため、課題 1 の時点で課題 2 も見越した設計を行った。

### 課題 2 を見越した設計

課題 1 の要件（連絡帳の記載・閲覧・既読処理）だけでなく、将来的な拡張を見据えて以下を設計した。

**ロールベース権限管理**: 担任だけでなく、学年主任・校長も使えるよう 5 ロールを設計した。各ロールで見える情報の粒度を変え、担任は詳細、学年主任は学年全体、校長は学校全体を確認できるようにした。

**データモデル設計**: 連絡帳だけでなく、出席記録、担任メモ、既読管理など、将来必要になる機能を見越してデータモデルを設計した。

### 一番苦労したのはデザイン設計

技術的な実装は、AI や Web でフォローされるのでそこまで困らなかった。一番苦労したのは「デザイン設計」。

初めてアプリを作ったので、ただ機能を盛り込むだけでは使いづらく、DX 化する意味がないと考えた。この DX で何を実現したいか、どうやったらユーザーが使いやすい形で体験できるか、シンプルにしつつ、意味をどう持たせて処理するかを試行錯誤で何度も繰り返した。

### なぜ Terraform で本番環境を作ったか

私はインフラ経験があり、開発経験はなかった。他の候補者とどう差別化するかを考えたとき、ただデプロイ環境を作るだけではなく、クラウド実装 + IaC で実現することでアピールできると考えた。

また、IaC は自分が習得したいスキルだった。画面操作では設定ミスが起きるが、コード化することでミスや手作業が減る。アピールもできるし、自分の学習にもなると考え、Terraform を選択した。

**補足資料**: [TECHNICAL_SPECIFICATION.md](./TECHNICAL_SPECIFICATION.md)、[PRODUCTION_DEPLOYMENT.md](./PRODUCTION_DEPLOYMENT.md)

## 2. 課題 2 のプレゼンテーション

### ヒアリング結果の解釈

課題 2 では、担任・生徒・学年主任へのヒアリング結果を元に、具体的な改善機能を実装した。

**担任の課題**:

- 「時系列でこの生徒の状態がどうなってるか把握したい」
- 「全体感も把握できたらいいな」
- 「担任間で共有したい生徒の状況をタイムリーに共有したい」

**生徒の課題**:

- 「毎日『体調：ふつう』って書いてるかも。数字で表した方がいいのかな?」

**学年主任の課題**:

- 「問題が起こりやすいクラスに早めに気づけるようになるといいな」

### Inbox Pattern による優先度別分類

担任が重要度が高いものから対応できるよう、生徒を 7 カテゴリに自動分類した。時間がない中で全生徒のページを開いて確認する作業を削減するため、カード化して、連絡帳の内容を一覧で確認し、既読・タスク化を処理できるようにした。

| カテゴリ           | 優先度 | 説明                                               |
| ------------------ | ------ | -------------------------------------------------- |
| P0: 重要           | 最高   | 3 日連続メンタル低下、メンタル ★1（即時対応）      |
| P1: 要注意         | 高     | メンタル ★2（注意が必要）                          |
| P1.5: 要対応タスク | 高     | 既読済みで対応記録あり（保護者連絡、個別面談など） |
| P2-1: 未提出       | 中     | 今日の連絡帳未提出                                 |
| P2-2: 未読         | 中     | 提出済みだが未確認                                 |
| P2-3: 反応待ち     | 低     | 既読済みだが反応未選択                             |
| P3: 完了済み       | 低     | 既読・反応済み                                     |

### 早期警告システム（3 日連続メンタル低下検知）

担任の力量により、対応が必要なのに対応できず、問題を抱え込んでしまうケースを防ぐため、メンタルで問題がある状態が 3 日続いたとき、学年主任に自動通知する仕組みを作った。起きていることが自動的にオープンになり、早期発見・早期対応が可能になる。

### 時系列推移表示

担任のヒアリング「時系列で把握したい」に対応するため、各カードに過去 3 日分の体調・メンタル推移を表示した（例: 体調: ★4 → ★3 → ★2 / メンタル: ★3 → ★2 → ★1）。担任は推移を見て「メンタルが下がり続けている」ことに気づける。

### 担任メモの共有機能

担任のヒアリング「タイムリーに共有したい」に対応するため、担任メモを学年内で共有できる機能を実装した。会議の場を待たずに、気づいたときに情報共有できる。

### 数値化（★1-5）

生徒のヒアリング「数字で表した方がいいのかな?」に対応するため、体調・メンタルを ★1-5 で表現した。これにより、推移の変化が一目で分かる。

## 3. 課題・インターンの感想

### 今回の課題で得たもの

初めて、インフラだけではなく、フルスタックでアプリを作りきった。「やった！」という達成感がある。

実装をする技術も大事だが、今後 AI がより発達する中で、より重要になるのは「このアプリで実現したいことは何か？その背景は？」と考える力だと感じた。ある程度の知識があれば、AI のおかげで機能を実装するハードルはそこまで高くない。

### 一番苦労したこと

今回の課題で一番苦労したのは、UX・デザインの部分。知識がなかったので、何度も試行錯誤してよりよくする努力を繰り返した。

今回の経験を通じて、アプリを作る楽しみと、今後重要になるシステム設計のデザイン力を学ぶ必要があると感じた。

### インターンへの期待

今回、自分の純粋な技術力を超えた実装ができた。作ったシステムを、今後の自分の学習環境として活用したい。「これはどういう意味だったのか」「運用はこう自動化しよう」など、技術の理解を深める環境にしたい。また、新しいことを試してみる環境にもしたい。

インターンという目的のおかげで作りきれたので、作ったあとも最大限自分の成長に利用したいと思う。
